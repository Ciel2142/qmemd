# LLM-agent memory: prior-art research for qmemd's open weak spots

- **Date:** 2026-06-27
- **Method:** multi-agent workflow `qmemd-memory-prior-art` (run `wf_1c1fd265-c5c`) — 1 codebase-grounding agent + 6 web/GitHub finders + 1 synthesis. 8 agents, ~546k tokens, 120 tool calls, 118 unique source URLs.
- **Purpose:** durable record of how other LLM-agent memory systems solve the 5 weak spots a critic + review surfaced in qmemd, feeding the `beacon→injection` spec.

## The 5 weak spots under study

| ID | Weak spot |
|----|-----------|
| **W1** | **Auto-recall trigger** — recall is pull-only/agent-initiated; nothing decides "retrieve now." Snapshot is a static partial push; beacon only nudges. |
| **W2** | **Discipline-dependence** — correctness depends on the LLM remembering to call recall; machine can only nag via prose. |
| **W3** | **Rule-surface bloat** — recall rules are several long paragraphs; rule fatigue (every fix-by-adding-prose raises next-miss odds). |
| **W4** | **Staleness / decay** — facts rot; `review_by`+`reviewed` impose a manual maintenance tax; nothing auto-expires. |
| **W5** | **Dedup / conflict** — near-dups + contradictions are surfaced, never resolved; an undisciplined agent can `--force` past. |

## Part 1 — qmemd current memory-delivery map (grounding)

# qmemd memory-delivery map — every mechanism that reaches the agent, and where each falls short

All paths are filesystem/lex-only except hybrid recall. Two of three live delivery surfaces (snapshot, beacon) are **push-on-a-schedule**; only recall is content-bearing, and it is **pull-only**.

## 1. Session snapshot — `recallSession()` (`src/engine.ts:511`)
**What it injects** (header `## Memory (qmd)`, then, in priority order):
- **user + feedback** facts — full body via `withBody` (`engine.ts:658`). The only "exhaustive" lanes, but still filtered by `active` (not superseded, `:521`) and `onPlat` (host OS, `:516`).
- **pinned** project/reference one-liners, gated to scope (`:553`).
- **project** facts: non-pinned, in-scope, newest-first, **sliced to `projectLimit` (default 5 / `$QMEMD_SESSION_PROJECT_LIMIT`)** — `inScope().slice()` `:560-564`.
- **reference** facts: same slice `:569-570`.

**What it slices/hides:**
- **Project scope gate** `inProject` (`:552`): only `frontmatter.project === cwd-basename || "global"`. Other repos' facts never appear.
- **Recency slice**: project/reference truncated to 5; the rest is *counted, not shown* via `footerLine` (`:599`) → `"N project facts for <proj> (5 shown, K more) — qmemd list…"`, plus `Unshown tags:` histogram (`:722`), platform-hidden footer (`:603`), unreadable footer (`:734`), `(N more — qmd recall to see)` (`:739`).
- **Byte budget** default 2000 / `$QMEMD_SESSION_BUDGET` (`:513`); user/feedback bodies greedily fill it, footers are reserved-then-best-effort.

**When it runs:** once at session start, via three entry points, each doing `gitPullFfOnly` first:
- CLI `qmemd recall --session` (`src/cli/qmemd.ts:287-301`, pull at `:288`), wrapped as SessionStart `hookSpecificOutput.additionalContext`.
- MCP `recall {session:true}` (`src/mcp/server.ts:239-253`, pull `:245`).
- REST `/recall {session:true}` (`server.ts:611-616`, pull `:614`).

**Gap → W1/W3:** It is a static, time-zero, **partial** push — sliced (5) + budget-capped + recency-ordered, with **zero relevance to the task the agent is about to do**. The footers (`completenessFooter` idiom) are the entire mitigation for partialness, and they *only work if the agent reads them and chooses to `recall`/`list`* — i.e. they convert a silent miss into another discipline-dependent rule (W3 rule-surface). Nothing here fires later in the session.

## 2. Beacon — `runBeacon()` (`src/beacon.ts:156`)
**Trigger:** PreToolUse on **Bash only** (`:159`). Throttle `decideBeacon` (`:28`): fires on **repo pivot** (`prev.repo !== repo`) OR every **`everyN` Bash calls** (default **20** / `$QMEMD_BEACON_EVERY`, `cli/qmemd.ts:495`).
**What it emits:** `formatBeacon` (`:64`) → repo name, `total` count, **tag histogram only** (`formatTagHistogram(ov.tags)`), and the hint `→ recall before diagnosing, e.g. qmemd recall "<proj> <topic>"`. Corpus scan (`projectOverview`, `engine.ts:2300`) is restricted to `["project","reference"]` and runs **only on a firing call** (`:179`).
**Does it inject facts? NO.** It carries the **tag shape and a count**, never any fact body or description. It is a *nudge to call recall*, explicitly fail-open and path-free (`qmemd-81n`).
Companion: `runWriteBeacon` (`:195`, Stop hook) — capture nudge, **OFF unless `QMEMD_WRITE_BEACON` truthy** (`cli/qmemd.ts:508`); also injects no memory.

**Gap → W1/W2/W3:** The beacon is the closest thing to an "auto-retrieve" trigger and it deliberately stops short of retrieving — it prints "you have N memories, go recall" and relies on the agent to act. Throttled to ~1/20 Bash calls, Bash-only (no nudge before Edit/Read/Write/Task), and content-free. It *adds* a rule ("recall before diagnosing") rather than removing the discipline requirement.

## 3. Recall — `recallQuery` / `recallQueryWithStatus` (`src/engine.ts:2034` / `:1782`)
**Invocation: manual only.** The only callers are agent-initiated: CLI `recall <query>`, MCP `recall` tool (`server.ts:219-305`), REST `/recall`. **There is no auto path** — no hook, no PreToolUse, nothing keyed on "agent is about to diagnose/answer" that calls `recallQuery`. The snapshot and beacon never call it either.

**The capability gate is a search-MODE selector, not a when-to-fire trigger — confirmed.** `shouldAutoResolve` / `autoRecallMode` / `resolveExplicitMode` (`src/capability.ts:128/134/23`) only ever return a `RecallMode` = `"lex" | "hybrid"`. In the handler (`server.ts:266-277`) this runs **after** the agent has already decided to call recall, and merely sets `effectiveLexOnly` (lex on a CPU-only host, hybrid on GPU). It selects the *algorithm*, not the *moment*. It does nothing for W1.

**Gap → W1/W2 (squarely):** Correctness depends entirely on the LLM deciding to call recall at the right instant. The machine cannot force it; the snapshot + beacon can only advertise that more exists. This is the core open weakness — no component decides "retrieve now."

## 4. Reindex — auto on writes, manual on edits
- **Auto:** `reindexMemory` (= `store.update`, `engine.ts:931`) runs inside `remember`, `forget` (`:2326`), `markReviewed` (`:2263`). Hybrid recall *also* self-heals **pending vectors** via the lazy embed barrier (`recallQueryWithStatus:1811-1842`) — but that only embeds *already-indexed* docs.
- **Manual:** out-of-band file edits (hand-edit a `.md`) are **not** picked up by a bare `recall` (lex index is stale); requires `qmemd reindex` then `qmemd embed` (`cli/qmemd.ts:538`; called out in CLAUDE.md).

**Gap → W4-adjacent:** editing a fact in place imposes a manual reindex tax; until then recall serves stale lex content with no warning.

## 5. Staleness — `staleFacts()` (`src/engine.ts:2162`), surface-only
- Computes two lanes: **due** (explicit `review_by ≤ today`, or implicit per-type window past — `ttlDefaultDays`: project 90d, reference 180d, user/feedback durable) and **never-reviewed backlog** (decay-prone, not yet due, sliced to `limit`).
- Surfaced only via `qmemd stale` and `qmemd doctor`. **`recall` ignores `review_by` entirely** (`:2158`); **nothing auto-expires**.
- Reset clock with `qmemd reviewed` → `markReviewed`/`resolveReviewedDate` (`:2250/:2231`), which forward-sets `review_by` and deliberately **does not bump `updated`** (keeps content-age honest). `review_by: never` = durable exempt (`:2176`).

**Gap → W4 (squarely):** facts rot but the system never mutates or drops them; staleness is visible only when the agent *chooses* to run `qmemd stale`, and clearing it is a manual `reviewed`/`replace`/`supersedes`/`forget` per fact — a pure maintenance tax with no automation.

## 6. Dedup / conflict — three-tier in `remember()`, never auto-resolved
- **Tier-1** exact-slug (file existence). **Tier-2** FTS BM25 > `DEDUP_SCORE_FTS` (`1e-6`, `engine.ts:766`). **Tier-2.5** model-free token-set near-dup: `nearDuplicate` (`:1117`) + contradiction classifier `classifyNearMatch` (`:1206`) over `identifiersConflict` (`:1087`), `antonymConflict` (`:1187`), `lowSimilarityConflict` (`:1258`).
- A likely contradiction routes to a **surfaced conflict** — `disposition:"conflict"` + `buildAuthorityComparison` (`:967`, via `authorityTier` `:1221`) — instead of a silent block or silent write.
- **The engine never auto-resolves:** the agent must pass `--replace` / `--force` / `--supersedes`. No automatic merge or winner selection.

**Gap → W5 (squarely):** near-duplicates and contradictions are *detected and surfaced*, then handed back to the agent's judgment. Resolution is 100% manual; an undisciplined agent can `--force` past the signal or never re-run, so the corpus can still accrete contradictions.

---
### One-line synthesis of the gap shape
Three delivery surfaces exist: **snapshot** (push, time-zero, partial slice, task-blind), **beacon** (push, throttled nudge, count+tags only, no facts), **recall** (pull, full content, but agent-triggered only). The capability gate tunes recall's *algorithm*, not its *timing*. So every content-bearing retrieval depends on the LLM choosing to call `recall` (W1/W2); the mitigations are all additional prose rules and footers (W3); and the curation layers (staleness W4, dedup/conflict W5) are surface-only — they inform the agent but never act, leaving maintenance and resolution as a manual tax.

## Part 2 — external prior art (6 angles, 45 findings)

### Angle: memory-libs

#### Claude Code UserPromptSubmit hook auto-injection (claude-mem / mem0 plugin)  ·  [W1, W2, W3]  ·  🟢 production

- **What:** Persistent-memory plugins for Claude Code that inject relevant memories deterministically on every user turn via the UserPromptSubmit lifecycle hook, instead of waiting for the agent to call a recall tool.
- **Mechanism:** A registered UserPromptSubmit hook fires synchronously AFTER the user submits but BEFORE the model processes the turn. The hook runs a semantic search against the memory store keyed on the just-submitted prompt (claude-mem hits /api/context/semantic, top-5, configurable via CLAUDE_MEM_SEMANTIC_INJECT_LIMIT) and returns the hits as hookSpecificOutput.additionalContext, which Claude Code splices into the model context for that turn. It is fail-open (returns exit SUCCESS on any worker error, 60s timeout) so a memory outage never blocks the turn. mem0's claude-code plugin uses the same userpromptsubmit.py pattern plus SessionStart/Stop/SessionEnd hooks to capture learnings. Retrieval is thus a deterministic middleware step, not an LLM decision.
- **Relevance to qmemd:** Highest-leverage, near-drop-in fix: qmemd already ships as a Claude Code plugin with a PreToolUse beacon hook (qmemd hook beacon) that only NUDGES. Promote it to a UserPromptSubmit hook that runs recallQuery(prompt) and emits hookSpecificOutput.additionalContext — turning the nag into actual injection with zero agent discipline (W1/W2) and letting the multi-paragraph recall rules in qmemd.md shrink to almost nothing (W3). Catches: (a) Anthropic has open bugs where additionalContext is silently dropped for some hook events — UserPromptSubmit injection works but PreToolUse/PostToolUse additionalContext is documented-but-unimplemented (issue #19909), so qmemd must use UserPromptSubmit specifically, not the existing PreToolUse beacon slot; (b) a recall now runs every turn = latency + token cost + relevance-precision risk (top-5 noise), so it needs a relevance floor and small K; (c) only covers the Claude Code surface, not the CLI/other-MCP-client surfaces.
- **Sources:** https://deepwiki.com/thedotmack/claude-mem/3.1.2-userpromptsubmit-hook; https://docs.mem0.ai/integrations/claude-code; https://github.com/0xtechdean/claude-code-mem0; https://github.com/anthropics/claude-code/issues/19909; https://alexandrekhoury.com/writing/superbrain-session-memory-claude-code

#### Supermemory Memory Router (transparent proxy / middleware injection)  ·  [W1, W2, W3]  ·  🟢 production

- **What:** A drop-in HTTP proxy that sits between the app and the LLM provider and automatically manages context — retrieving + injecting relevant memory and trimming old context — with no app code changes.
- **Mechanism:** You swap the LLM client base URL to https://api.supermemory.ai/v3/. The router intercepts each chat-completion request, runs RAG over the user's stored memories/profile, appends the most relevant memories to the outgoing prompt, strips unnecessary older context (claims up to 70% token savings), forwards the augmented prompt to the real provider (OpenAI/Anthropic/Gemini/Groq), and extracts/writes new memories ASYNCHRONOUSLY after the response so writes never block. Retrieval timing is entirely in the proxy layer — the agent is never asked to decide 'retrieve now'.
- **Relevance to qmemd:** Proves the strongest version of automatic retrieval: put memory I/O in the request path, not in the agent's tool-choice. qmemd can't literally proxy the model from inside an MCP server, but the architectural lesson maps onto the Claude Code hook (finding above) which is qmemd's available 'request path'. Borrow specifically: async write-after-response (qmemd already writes lex-first), and proxy-side context trimming. Catches: a true proxy requires owning the model endpoint (Supermemory is a hosted SaaS); it is opaque about conflict/forget policy and gives no per-request latency numbers; for qmemd the equivalent injection point is the hook, which is cheaper to adopt than re-architecting as a gateway.
- **Sources:** https://supermemory.ai/docs/memory-router/overview; https://deepwiki.com/supermemoryai/supermemory/8.5-memory-router-and-model-enhancement; https://blog.logrocket.com/building-ai-apps-mem0-supermemory/

#### mem0 LLM-driven ADD/UPDATE/DELETE/NOOP write pipeline  ·  [W5, W4]  ·  🟢 production

- **What:** On every memory write, mem0 auto-resolves dedup and contradiction by letting an LLM choose one of four operations against the existing neighbors, instead of surfacing the conflict to a human.
- **Mechanism:** add() extracts candidate facts, retrieves the top semantically-similar existing memories, and hands {candidate fact + retrieved neighbors} to an LLM via a function-calling interface that must return ADD (no equivalent exists), UPDATE (augment an existing memory), DELETE (an existing memory is contradicted by the new fact — auto-removed), or NOOP (no change). Runs in 300–800ms and is meant to run asynchronously after the user response. Retrieval at query time fuses semantic + BM25 keyword + entity signals via rank fusion into top-K.
- **Relevance to qmemd:** This is the auto-resolve path qmemd deliberately declined: qmemd's three-tier dedup SURFACES near-dups/contradictions (disposition:'conflict') and never mutates. mem0 shows the opposite design — DELETE silently retires the contradicted/stale fact (which also addresses W4 decay without a manual reviewed command). Borrowable as an OPT-IN 'auto-resolve' mode on remember (e.g. high-confidence contradiction → propose UPDATE/supersede). The catch is exactly why qmemd stayed conservative: the LLM occasionally DELETEs a still-correct fact or wrongly merges, and it costs an LLM call per write — whereas qmemd's contradiction classifier is intentionally model-free. Keep auto-resolve gated behind an authority/confidence threshold and write a .bak (doctor already does this).
- **Sources:** https://arxiv.org/abs/2504.19413; https://arxiv.org/html/2504.19413v1; https://github.com/mem0ai/mem0; https://docs.mem0.ai/core-concepts/memory-evaluation

#### Zep / Graphiti bi-temporal edge invalidation  ·  [W5, W4]  ·  🟢 production

- **What:** A temporal knowledge graph that handles conflict and staleness structurally: contradicting facts invalidate old edges by timestamp rather than deleting them, and retrieval returns only the currently-valid slice.
- **Mechanism:** Every fact is an edge carrying four timestamps: t_valid / t_invalid (when the fact was true in the world) and t_created / t_expired (when the system learned/retired it) — a bi-temporal model. Ingesting a new episode runs LLM entity+relationship extraction; when a new edge contradicts an existing one, Graphiti sets the old edge's t_invalid instead of discarding it, preserving history. Queries (hybrid semantic + keyword + graph traversal) filter to edges valid as-of-now (or as-of an arbitrary date), so stale facts auto-drop from results without ever being deleted and without a manual review pass.
- **Relevance to qmemd:** Directly attacks W4+W5 with no human maintenance tax: 'supersede, never delete; mark invalid; filter at recall.' qmemd already has supersedes / superseded_by / conflicts_with frontmatter but only uses them for display — it does NOT filter superseded facts out of recall, and review_by still requires a manual reviewed command. Borrow: (1) auto-exclude superseded_by facts from default recall (cheap, no graph needed); (2) add a valid_from/invalid_at pair so a contradicting remember stamps the old fact invalid and recall hides it by default but can answer 'as-of' queries. Catch: full Graphiti needs a graph DB (Neo4j/FalkorDB) + an LLM extraction call per episode — far heavier than qmemd's flat-markdown + lex index; take the temporal-validity pattern, not the graph.
- **Sources:** https://arxiv.org/abs/2501.13956; https://github.com/getzep/graphiti; https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/; https://help.getzep.com/graphiti/getting-started/overview

#### Letta (MemGPT) always-on core memory blocks + sleep-time compute  ·  [W1, W2, W4]  ·  🟢 production

- **What:** Letta splits memory into an always-injected in-context tier and tool-retrieved tiers, and runs a background agent during idle time to consolidate/dedup memory without blocking the conversation.
- **Mechanism:** Core memory = small character-bounded blocks that are COMPILED INTO EVERY context window automatically (no retrieval call) — the agent edits them via tools but never has to 'recall' them. Only the long tail — recall memory (full interaction history) and archival memory (external DB) — requires the agent to call a search tool. Sleep-time compute spins up a separate 'sleep' agent that reprocesses interaction history asynchronously during downtime, rewriting/reorganizing memory blocks into higher-quality, deduped state, decoupling memory maintenance from the live response loop.
- **Relevance to qmemd:** Two borrowable ideas. (1) Always-on hot block: qmemd's session snapshot is the analog but it is PARTIAL and fires only at session start; promote a tiny curated set (pinned + most-relevant project facts) into per-turn injection (via the hook) so the hot set never depends on the agent calling recall (W1/W2). (2) Sleep-time = a background 'qmemd consolidate' job (cron / SessionEnd hook) that re-embeds, runs dedup/supersession, and refreshes review_by automatically — turning W4's manual maintenance tax into an idle-time agent. Catches: always-on costs a fixed token budget every turn so the hot block must stay small and well-ranked; sleep-time needs orchestration + extra LLM compute and can entrench a bad merge if it auto-resolves (pair with the gated auto-resolve from the mem0 finding).
- **Sources:** https://www.letta.com/blog/agent-memory/; https://www.letta.com/blog/memory-blocks/; https://www.letta.com/blog/sleep-time-compute/; https://docs.letta.com/letta-code/memory

#### A-MEM (Zettelkasten agentic memory + memory evolution)  ·  [W5, W4]  ·  🔬 research

- **What:** Each new memory is auto-annotated and linked into a note network, and crucially the insertion can trigger updates to the attributes of the older notes it links to — so conflicts/refinements propagate instead of accumulating as silent duplicates.
- **Mechanism:** On write, an LLM constructs a structured note (contextual description, keywords, tags) from the raw memory, then finds semantically related historical notes and creates explicit links. 'Memory evolution': integrating the new note can rewrite the contextual representation/attributes of the linked existing notes, so the network continuously re-organizes (t-SNE shows tighter clusters than static stores). No fixed, predetermined schema of memory operations — structure emerges agentically.
- **Relevance to qmemd:** Shows a middle path between qmemd's 'surface conflict, do nothing' and mem0's 'LLM deletes': let a new fact UPDATE the metadata/links of related old facts rather than spawn a near-dup. qmemd already auto-tags and indexes; the cheap borrow is auto-linking new facts to existing ones by shared tags/tokens at write time (no LLM) to make dedup/supersession candidates visible and navigable. Catch: the full evolution step is LLM-per-insert and research-grade (NeurIPS 2025, no production hardening); auto-rewriting old facts' attributes risks the same silent-corruption problem qmemd's design avoids, so adopt the linking, not the unsupervised rewrite.
- **Sources:** https://arxiv.org/abs/2502.12110; https://github.com/WujiangXu/A-mem; https://openreview.net/forum?id=FiM0M8gcct

#### MemoryOS heat-based eviction + Memary recency/frequency ranking  ·  [W4, W1]  ·  🔬 research

- **What:** Two systems that decide what stays in (and gets injected from) memory by automatic usage-decay signals — heat/recency/frequency — rather than a human marking facts stale.
- **Mechanism:** MemoryOS mimics an OS: STM → MTM → LPM tiers with segment-paging of dialogue grouped by topic, and HEAT-DRIVEN eviction (frequency + engagement + recency) that automatically demotes/evicts cold content across tiers (~50% over baselines on LoCoMo). Memary tracks every extracted entity in a 'memory stream' and an Entity Knowledge Store that ranks entities by BOTH frequency (occurrence count) and recency and auto-passes the top-N into the context window on each turn as part of the response pipeline — injection is automatic, and memory updates write back to the Neo4j KG as the agent interacts.
- **Relevance to qmemd:** An alternative to qmemd's date-based review_by tax (W4): let access patterns drive staleness/relevance. qmemd already records created/updated and could cheaply log per-fact access/hit counts at recall time, then (a) auto-rank injection by recency+frequency like Memary's top-N and (b) flag never-accessed-and-old facts as decay candidates automatically instead of waiting on the reviewed command. Catches: heat/decay needs usage telemetry qmemd doesn't currently track (a new counter + storage), and both systems are tuned for conversational dialogue history, not durable curated project facts — so frequency must not silently evict a rarely-used-but-load-bearing fact (keep pinned facts exempt, mirroring review_by: never).
- **Sources:** https://arxiv.org/abs/2506.06326; https://github.com/BAI-LAB/MemoryOS; https://kingjulio8238.github.io/memarydocs/concepts/; https://github.com/kingjulio8238/memary

**Key takeaways:**

- The single biggest lever for qmemd's W1/W2 is moving retrieval OUT of the agent's tool-choice and INTO a deterministic middleware step. Two production patterns prove it: Supermemory's transparent LLM proxy, and (directly available to qmemd today) the Claude Code UserPromptSubmit hook that runs a semantic search on the submitted prompt and returns hookSpecificOutput.additionalContext before the model sees the turn. qmemd already has the hook plumbing (the PreToolUse beacon) but only nudges — upgrading to UserPromptSubmit injection is the highest-ROI change.
- Auto-injection also collapses W3 (rule-surface bloat): claude-mem/mem0 ship almost no prose telling the model 'remember to recall' because the hook does it unconditionally. If qmemd injects per-turn, the multi-paragraph recall discipline in qmemd.md can shrink to a sentence — every word removed reduces rule fatigue.
- Use UserPromptSubmit specifically, not PreToolUse, for injection: Anthropic has open issues (#19909) where additionalContext is silently dropped on PreToolUse/PostToolUse but works on UserPromptSubmit. Whatever qmemd builds must fail-open (exit 0, short timeout) like claude-mem so a memory outage never blocks a turn.
- On W5/W4 there's a clear design spectrum: qmemd today = surface-only (never resolve); mem0 = LLM ADD/UPDATE/DELETE/NOOP auto-resolve (can wrongly delete); Zep/Graphiti = bi-temporal invalidate-don't-delete (auditable, reversible). The Zep model is the safest borrow and fits qmemd's existing supersedes/superseded_by frontmatter — the cheap win is to auto-EXCLUDE superseded facts from default recall and stamp a contradicting write as invalidating the old fact, with an 'as-of' escape hatch.
- For W4 staleness without a manual maintenance tax, two automated alternatives to date-based review_by emerged: (a) background consolidation a la Letta sleep-time compute — a SessionEnd/cron 'qmemd consolidate' agent that re-embeds, dedups, and refreshes review_by during idle; (b) usage-decay ranking a la MemoryOS heat / Memary recency+frequency — let access counts, not calendar dates, drive what's surfaced and what's flagged stale. Both require new telemetry qmemd doesn't yet keep (access/hit counters), and both must exempt pinned facts.
- Letta's always-on core-memory blocks vs tool-retrieved archival is the cleanest framing of qmemd's gap: a small curated hot set should be injected every turn (qmemd's snapshot, but per-turn and ranked), while the long tail stays behind explicit recall. The win is that the agent never has to 'decide to recall' the hot set.
- Lower-relevance survey members for completeness: cognee (ECL extract→cognify→load into a vector+graph store; retrieval traverses the graph, still a search() call — no auto-trigger) and txtai (plain embeddings/SQLite semantic store — storage only, retrieval is an explicit query) add little on the auto-trigger axis. Memobase contributes a useful narrow idea: structured profile SLOTS mean an update overwrites a slot in-place (no near-dup) and a token/time buffer auto-flushes extraction — good for user-preference facts but not arbitrary project facts. Cipher (byterover) is auto-CAPTURE but retrieval is still an MCP tool (cipher_memory_search), so it does not solve W1.

### Angle: ide-agent-memory

#### Claude Code native context-injection hooks (SessionStart + UserPromptSubmit → additionalContext)  ·  [W1, W2, W3]  ·  🟢 production

- **What:** Claude Code's own hook system can deterministically push text into the model's context window at session start and on every user prompt, with no agent action required. This is the platform primitive that 'zero-discipline' memory tools build on.
- **Mechanism:** SessionStart and UserPromptSubmit hooks are the two 'context-injecting' lifecycle events: whatever the hook emits on stdout (plain text) or via hookSpecificOutput.additionalContext is inserted as real conversation context BEFORE the model sees the turn. As of Claude Code 2.1.0 this injection is silent (no user-visible message). Crucially, the docs/community frame this as 'deterministic rather than advisory' — unlike CLAUDE.md, which the model may ignore, hook-injected context is always present. UserPromptSubmit fires immediately on submit, before the model processes the prompt, and can only ADD context (not replace the prompt). The hook is just a shell command, so it can run any retrieval (DB query, vector search) and feed results back deterministically.
- **Relevance to qmemd:** qmemd ALREADY ships a hook entrypoint (`qmemd hook beacon`, a PreToolUse nudge). The same plumbing can register a SessionStart hook (qmemd already has `recall --session` snapshot logic) and/or a UserPromptSubmit hook that runs recallQuery and emits hookSpecificOutput.additionalContext — converting recall from agent-optional (W1/W2) into a deterministic middleware step. This is the single highest-leverage borrow: it deletes the 'agent must remember to call recall' dependency entirely. Catch: UserPromptSubmit fires on EVERY prompt, so qmemd must keep it inside a tight latency budget (lex-only or cached vectors, like the existing 5s git timeout) and bound output (top-K + progressive disclosure) or it re-creates W3 (bloat) and adds per-turn latency. Also the query is now the raw user prompt, not the curated topic string an agent passes today — so retrieval quality must survive noisier queries.
- **Sources:** https://code.claude.com/docs/en/hooks; https://docs.anthropic.com/en/docs/claude-code/hooks; https://www.mindstudio.ai/blog/session-start-hooks-claude-code-force-context

#### mann1x/claude-hooks — deterministic memory recall pipeline on UserPromptSubmit (HyDE + attention decay + dedup + progressive disclosure)  ·  [W1, W2, W3, W4, W5]  ·  🧪 experimental

- **What:** An open-source Claude Code hook set that does exactly what qmemd's W1 is missing: on every prompt it deterministically retrieves, ranks, dedups and injects memories from a vector store — no agent recall call. It is effectively a reference implementation of the qmemd gaps.
- **Mechanism:** The UserPromptSubmit hook runs a multi-stage pipeline per prompt: (1) HyDE query expansion — generate a hypothetical answer via a local Ollama model first, then search, 'dramatically improving recall quality' for short/noisy prompts; (2) parallel fan-out recall across providers (Qdrant, Memory KG, pgvector, sqlite-vec) by cosine similarity; (3) attention/temporal decay re-rank — fades old memories, boosts frequently-recalled ones, tracked in JSON access state with a configurable decay_halflife_days; (4) dedup via cosine dedup_threshold before injection; (5) progressive disclosure — inject only first line + char count per memory (~3-5x less context); (6) on Stop, 'instinct extraction' detects error→edit patterns and auto-writes reusable markdown under ~/.claude/instincts/. A SessionStart hook re-injects full memory after Claude Code compacts context. Entire pipeline is fail-open (always exits success).
- **Relevance to qmemd:** This is the closest existing analogue to 'qmemd-with-the-weak-spots-fixed' and worth reading directly. Borrowable pieces: HyDE-expand the raw prompt before lex/hybrid search (compensates for prompt≠good-query in the hook model); attention-decay re-rank tied to recall-access counts — this attacks W4 (staleness) WITHOUT the manual review_by/`reviewed` tax, by letting usage frequency, not a human, set salience; progressive-disclosure injection (first line + count) is a clean W3 answer; cosine dedup gate is a W5 lever. Catch: HyDE needs a local LLM (Ollama) on the hot path — qmemd deliberately avoids model load on writes/snapshots, so this would be a new dependency and latency source; attention-decay needs per-fact access-state (a new sidecar with write-amplification + concurrency to manage). Experimental/niche project, so treat as design reference, not a dependency.
- **Sources:** https://github.com/mann1x/claude-hooks

#### claude-mem — async capture worker + silent SessionStart injection + progressive-disclosure index  ·  [W1, W2, W3]  ·  🟢 production

- **What:** A widely-used (46k★) Claude Code memory plugin whose injection is fully deterministic and model-free: it captures tool activity in the background and silently injects a compressed index at SessionStart. The agent never decides to retrieve.
- **Mechanism:** Uses all five Claude Code lifecycle hooks. PostToolUse fires after every tool call and does a non-blocking HTTP POST (~8ms) to a background Bun worker that AI-compresses 1k-10k-token tool outputs into ~500-token semantic 'observations' (typed: decision/bugfix/feature/refactor/discovery/change), stored in local SQLite FTS5 + Chroma (hybrid lex+vector). The SessionStart hook then DETERMINISTICALLY queries recent summaries (last 10) + observations (default 50) and injects them via hookSpecificOutput.additionalContext — silently since CC 2.1.0. Key W3 trick: it injects a 'progressive disclosure index' (titles, timestamps, token counts) rather than full bodies; the model pulls full content on demand via MCP search tools only when needed. Capture is decoupled from compression so hooks stay <1s while AI compression (5-30s) runs async.
- **Relevance to qmemd:** Two concrete borrows. (1) Wire qmemd's existing `recall --session` snapshot to an actual SessionStart hook emitting additionalContext, so the snapshot is deterministically injected instead of described in prose rules (kills W2 for the session-start case). (2) Adopt the progressive-disclosure INDEX pattern: inject an index of fact names/types/tags/dates + 'recall <tag> for full text', not the fact bodies — directly shrinks the injected payload (W3) while keeping everything reachable via qmemd's existing MCP recall/get tools. Catch: claude-mem auto-captures EVERYTHING (an observation per tool call) — the opposite of qmemd's curated-fact philosophy and its three-tier dedup discipline. Adopting the index/SessionStart-injection format is cheap and aligned; adopting auto-capture would change qmemd's nature and reintroduce the noise qmemd's curation avoids.
- **Sources:** https://docs.claude-mem.ai/hooks-architecture; https://github.com/thedotmack/claude-mem

#### Mem0 — two-phase extract/update with LLM-decided ADD / UPDATE / DELETE / NOOP auto-resolution  ·  [W5, W4]  ·  🟢 production

- **What:** Mem0 is the explicit counter-design to qmemd's 'surface conflicts, never auto-resolve' stance: it runs an LLM that decides, per candidate fact, whether to add, enrich, delete-contradicting, or no-op against existing memories — automated dedup AND conflict resolution.
- **Mechanism:** Per message pair, Phase 1 (Extraction) uses an LLM to distill salient candidate facts from the latest exchange + a rolling summary. Phase 2 (Update) retrieves vector-similar existing memories and uses an LLM TOOL-CALL to emit one of four operations per candidate: ADD (fact is genuinely new), UPDATE (enrich/extend an existing memory), DELETE (remove a memory the new fact contradicts), or NOOP (no change) — this is what keeps the store consistent and non-redundant automatically. The graph variant (Mem0g) adds a Conflict Detector flagging contradictory nodes/edges and an LLM Update Resolver that adds/merges/INVALIDATES/skips — and critically marks contradicted relationships as INVALID rather than deleting them, enabling temporal reasoning (what was true when).
- **Relevance to qmemd:** Directly addresses W5 (and W4 via temporal invalidation). qmemd already has the hard part — a contradiction classifier that detects differing identifier / polarity / antonym and surfaces disposition:'conflict' with an authority comparison. Mem0 shows the missing last step: an opt-in 'auto-resolve' mode where that classifier emits an UPDATE/supersede instead of only surfacing. The cleanest, lowest-risk borrow is Mem0g's 'invalidate, don't delete' — it maps almost 1:1 onto qmemd's EXISTING supersedes/superseded_by frontmatter, giving auto-resolution that's reversible and auditable (a contradicted fact is marked superseded, not destroyed). Catch: Mem0's resolution leans on an LLM judge per candidate — cost, latency, and nondeterminism — which is exactly what qmemd's deliberately model-free, deterministic write path avoids. Full auto-resolve also contradicts qmemd's stated design (engine never auto-resolves); the defensible middle ground is the supersede-on-high-confidence-contradiction path gated behind an explicit flag.
- **Sources:** https://docs.mem0.ai/core-concepts/memory-operations/add; https://arxiv.org/abs/2504.19413; https://memo.d.foundation/breakdown/mem0

#### Cursor Rules type system (Always / Auto-Attached glob / Agent-Requested / Manual) + sidecar Memories observer  ·  [W1, W3]  ·  🟢 production

- **What:** Cursor splits memory/instruction injection into four tiers by HOW activation is decided, and auto-WRITES memories via a separate observer model. The glob 'Auto-Attached' tier is a strong, discipline-free answer to rule-surface bloat (W3).
- **Mechanism:** Four rule types: (1) Always — alwaysApply:true, injected into every session unconditionally; (2) Auto-Attached — a `globs:` pattern (e.g. src/components/**/*.tsx) makes the rules engine inject the rule ONLY when a matching file is in context, so irrelevant rules never consume tokens; (3) Agent-Requested — rule has a `description` but no globs; the model reads descriptions and pulls the rule when it judges it relevant (selection delegated to the model); (4) Manual — only on @-mention. Separately, a 'sidecar' observer model watches the chat and proposes memories to save (user approves/rejects); saved memories become auto-managed rules that the rules engine injects on the first prompt of every new chat — no re-explaining, no agent recall call.
- **Relevance to qmemd:** The glob Auto-Attach tier is the key W3 idea: scope injection to the files/paths actually in play so only relevant facts inject, instead of one always-on wall of rules. qmemd facts already carry tags/project; adding optional path-glob scope metadata would let a SessionStart/PreToolUse hook inject only facts whose scope matches the touched files — turning the monolithic recall-rules paragraph into targeted, file-triggered injection. The sidecar observer is also a W1 write-side idea: a separate pass proposes 'remember this' so capture doesn't depend on the working agent's discipline either. Catch: qmemd facts are topical, not file-scoped, so glob-attach needs new metadata and a notion of 'files in context' that the CLI path doesn't have today; and the Agent-Requested tier is explicitly the W2 trap (still defers to the model) — only the Always + glob tiers are truly discipline-free.
- **Sources:** https://cursor.com/docs/context/rules; https://cursor.com/docs/context/memories

#### GitHub Copilot — copilot-instructions.md (always-injected) + *.instructions.md with applyTo glob scoping  ·  [W1, W2, W3]  ·  🟢 production

- **What:** The simplest possible zero-discipline mechanism, and a clean validation of the 'one always-injected file + glob-scoped extras' pattern. No agent decision, no per-user opt-in.
- **Mechanism:** .github/copilot-instructions.md is automatically prepended to EVERY chat request — 'no opt-in required, no per-user setup', committed once and every contributor gets it. For scoping, *.instructions.md files use a frontmatter `applyTo:` glob: with a pattern, the file is auto-injected whenever a matching file is in context; WITHOUT applyTo, the file is inert (does nothing until manually attached). So the glob is literally the on/off switch — the system, not the model, decides activation by file path.
- **Relevance to qmemd:** Validates that a tiny, static 'always-injected core + glob-scoped extras' beats long prose rules for the high-value fixed set. qmemd's pinned + user + feedback facts (the durable, always-relevant tier) could be compiled into a single always-injected block via a hook, exactly mirroring copilot-instructions.md — that removes those from the brittle recall-rules prose (W3) and makes them present unconditionally (W1/W2). The applyTo-glob idea reinforces finding 5's path-scoping. Catch: it's purely static — no ranking, no retrieval, no staleness handling. It solves W1/W2/W3 only for a small fixed set; the large, evolving project/reference corpus still needs ranked recall on top, which is precisely why qmemd can't be only this — but it can be this for the pinned/user/feedback tier.
- **Sources:** https://code.visualstudio.com/docs/agent-customization/custom-instructions; https://docs.github.com/en/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot

#### Cline / Roo-Code 'Memory Bank' — the discipline-dependent anti-pattern (cautionary contrast)  ·  [W1, W2]  ·  🟢 production

- **What:** A hugely popular structured-markdown memory approach that, despite the polish, has EXACTLY qmemd's W1/W2/W3 weakness: it relies on a long custom-instructions prompt telling the agent to read memory files, with no enforcing mechanism.
- **Mechanism:** Memory Bank is 'a methodology, not a fixed feature': a set of required markdown files (projectbrief.md, productContext.md, activeContext.md, systemPatterns.md, techContext.md, progress.md). The load mechanism is a custom-instruction sentence — 'Cline MUST read ALL memory bank files at the start of EVERY task — this is not optional' — i.e. enforced purely by prose, exactly as qmemd's recall-rules are. Roo-Code's variant adds a MANUAL trigger phrase ('UMB' / 'Update Memory Bank') the user types to checkpoint context before the window clears. There is no hook, no middleware: correctness depends on the model obeying the instruction and the user remembering to say the magic words.
- **Relevance to qmemd:** Included as the explicit cautionary contrast to findings 1-3: this is the most-adopted markdown-memory pattern and it does NOT solve the trigger problem — it just writes the trigger as a longer, more emphatic prompt ('this is not optional', ALL CAPS), which is the W3 rule-fatigue spiral qmemd is already in. The lesson is directive: do NOT try to fix qmemd's missed recalls by adding more prose to the recall rules; that path is proven brittle even when the memory store itself is excellent. Move the trigger out of the prompt and into a SessionStart/UserPromptSubmit hook (findings 1-3). Catch: none to borrow on the injection side — its value to qmemd is precisely as negative evidence that prompt-enforced loading caps out, justifying the hook investment.
- **Sources:** https://docs.cline.bot/prompting/cline-memory-bank; https://github.com/GreatScottyMac/roo-code-memory-bank; https://docs.windsurf.com/windsurf/cascade/memories

**Key takeaways:**

- The single decisive fix for W1/W2 is platform-level: stop describing recall in prose and move it into a SessionStart and/or UserPromptSubmit hook that emits hookSpecificOutput.additionalContext. This is deterministic ('always present') vs CLAUDE.md/rules ('model may ignore'). qmemd already ships a hook (`qmemd hook beacon`, PreToolUse) and a `recall --session` snapshot — the gap is wiring recall output through additionalContext instead of nagging the agent to call recall.
- There is a working reference implementation of 'qmemd with the weak spots fixed': mann1x/claude-hooks does deterministic per-prompt recall with HyDE expansion, attention/temporal decay ranking (W4 without a manual review tax), cosine dedup (W5), and progressive-disclosure injection (W3) — all fail-open. Read it as a design blueprint, not a dependency (it needs Ollama on the hot path, which conflicts with qmemd's model-free write path).
- W3 (bloat) is solved by injecting an INDEX, not bodies: claude-mem and mann1x both inject titles/types/dates/token-counts with full content fetched on demand via MCP search/get. qmemd already has MCP recall/get — switching the injected payload to a progressive-disclosure index is low-risk and directly shrinks per-turn tokens.
- W3 also wants SCOPING, not just compression: Cursor's glob Auto-Attach and Copilot's applyTo inject a rule only when a matching file is in context. qmemd facts are topical, not file-scoped; adding optional path-glob scope metadata would let a hook inject only facts relevant to the files being touched, replacing the monolithic recall-rules paragraph with targeted injection.
- W4 (staleness) has two non-manual designs to borrow: (a) usage-based attention decay (mann1x) where recall-frequency, not a human's `reviewed` command, sets salience; (b) Mem0g's 'invalidate, don't delete' temporal model, which maps almost 1:1 onto qmemd's existing supersedes/superseded_by frontmatter for reversible, auditable decay.
- W5 (dedup/conflict) is where qmemd most deliberately diverges from the field: Mem0 runs an LLM ADD/UPDATE/DELETE/NOOP judge to auto-resolve, which qmemd's design explicitly refuses. The defensible middle ground: keep qmemd's deterministic, model-free write path, but add an opt-in 'supersede on high-confidence contradiction' that reuses qmemd's existing contradiction classifier to mark a fact superseded (not deleted) — auto-resolution without an LLM on the write path.
- Negative evidence matters: Cline/Roo-Code Memory Bank is the most popular markdown-memory pattern and it has qmemd's exact W1/W2 flaw — load is enforced only by an emphatic prose instruction ('MUST read ALL files at start of EVERY task'). Its popularity does NOT validate prose-enforced loading; it validates the need to move the trigger into a hook. Do not fix misses by lengthening the rules.
- Two activation philosophies recur across every tool: (1) deterministic/system-decided injection (always-files, glob auto-attach, SessionStart/UserPromptSubmit hooks) = zero agent discipline; (2) model-decided retrieval (MCP recall tools, Cursor 'Agent-Requested' rules, basic-memory build_context) = same W2 trap qmemd is in today. qmemd currently lives entirely in camp 2 (MCP recall + prose rules); every robustness win in the survey comes from shifting the durable/high-value tier into camp 1.

### Angle: when-to-retrieve

_when-to-retrieve: adaptive / automatic retrieval techniques that decide "retrieve now?" without the agent explicitly asking — and whether each removes the agent-discipline dependence that qmemd's manual `recall` currently leans on._

#### Self-RAG (reflection tokens — retrieve-on-demand baked into decoding)  ·  [W1, W2]  ·  🔬 research

- **What:** Fine-tunes the generator to emit special 'reflection tokens': a Retrieve token that decides per-segment whether to fetch, plus critique tokens (ISREL relevance, ISSUP support, ISUSE usefulness) that score retrieved passages and its own output.
- **Mechanism:** Retrieval becomes a learned decode-time action, not an external tool call. During generation the model predicts Retrieve={yes/no/continue} at each step; on 'yes' it pulls passages, then emits ISREL/ISSUP/ISUSE tokens to grade and gate what it uses. The 'when to retrieve' decision is moved inside the model weights, so no prompt rule and no agent willpower is involved — and a threshold over the critique-token probabilities lets you tune retrieval frequency at inference.
- **Relevance to qmemd:** It is the cleanest answer to W1/W2 in principle (the machine itself decides), but the catch is fatal for qmemd's architecture: it requires fine-tuning the generator with custom tokens. qmemd sits behind a closed Claude model via MCP and cannot bake reflection tokens into Claude. The transplantable idea is conceptual only — qmemd would need a separate small local model to emit the 'retrieve-now' signal, since it can't modify the agent doing the generating.
- **Sources:** https://arxiv.org/abs/2310.11511; https://selfrag.github.io/; https://aclanthology.org/forum?id=hSyW5go0v8

#### FLARE (Forward-Looking Active REtrieval — confidence-threshold trigger)  ·  [W1, W2]  ·  🔬 research

- **What:** Generates the next sentence speculatively with no retrieval; if any token in that draft sentence has probability below a threshold θ, it treats low confidence as a knowledge gap, retrieves using the draft sentence as the query, and regenerates.
- **Mechanism:** Token log-probabilities are the trigger signal. θ=0 never retrieves, θ=1 retrieves every sentence; the authors found firing on ~40-80% of sentences is optimal. Retrieval is automatic and continuous (mid-generation, not just at the start) and keyed to the model's own uncertainty rather than to an agent remembering to ask.
- **Relevance to qmemd:** Validates 'uncertainty = the trigger' over 'agent discipline = the trigger.' Catch: FLARE needs the generator's token logprobs, which qmemd (an MCP layer over markdown) never sees — Claude's logprobs aren't exposed. qmemd cannot implement FLARE against Claude directly; it would need a local draft model to produce the confidence signal. Still, the principle reframes qmemd's whole problem: stop asking the agent to self-report 'I should recall' and instead derive a signal it cannot fake.
- **Sources:** https://arxiv.org/abs/2305.06983; https://github.com/jzbjyb/FLARE; https://aclanthology.org/2023.emnlp-main.495/

#### Adaptive-RAG (external complexity classifier routes the query)  ·  [W1, W2, W3]  ·  🔬 research

- **What:** A small dedicated classifier (T5-Large) predicts each query's complexity and routes it to one of three paths: no-retrieval, single-step retrieval, or multi-step iterative retrieval.
- **Mechanism:** The retrieve/don't-retrieve decision is offloaded to a cheap external model trained on auto-labeled data (labels derived from whether smaller/larger models actually answered correctly). The agent never decides; a router classifier does, before the expensive generator runs. This is the canonical 'gating/routing classifier' pattern and is described in 2026 practitioner writeups as the emerging production default for complexity routing.
- **Relevance to qmemd:** Most architecturally compatible of the trained approaches: the classifier is OUTSIDE the generator, exactly where qmemd lives. qmemd could run a tiny local gate (it already loads embeddinggemma-300M) over the incoming prompt to decide 'fire recall vs not,' replacing several paragraphs of recall rules (W3) with a deterministic call. Catch: needs labeled training data ('was a memory hit useful here?') qmemd doesn't yet collect, and it routes by *complexity*, not by 'is there a relevant fact in store' — qmemd would instead want a 'memory-relevance' gate (closer to a cheap lex/vector pre-probe of the corpus than a complexity classifier).
- **Sources:** https://arxiv.org/abs/2403.14403; https://aclanthology.org/2024.naacl-long.389/

#### TARG — Training-Free Adaptive Retrieval Gating (+ the 'uncertainty beats complex pipelines' benchmark)  ·  [W1, W2]  ·  📄 paper

- **What:** A single-shot, model-agnostic gate that decides whether to retrieve from a short no-context draft: it reads prefix logits, computes a lightweight uncertainty score (mean token entropy, top1/top2 margin, or small-N variance), and retrieves only when the score crosses a threshold. No fine-tuning, no auxiliary head.
- **Mechanism:** Generate ~tens-to-hundreds of draft tokens with no retrieval, score uncertainty from those logits, gate. vs Always-RAG it cuts retrieval 70-90% while matching EM/F1 and staying near Never-RAG latency; the margin signal is the robust default on modern instruction-tuned models. Backed by the ACL 2025 survey of 35 adaptive-retrieval methods which found simple uncertainty estimators often beat complex adaptive pipelines on efficiency + self-knowledge at comparable QA — i.e. you do NOT need a heavyweight trained router.
- **Relevance to qmemd:** The most directly actionable 'when-to-retrieve' mechanism for qmemd because it is training-free and model-agnostic — qmemd could run a small local draft model to compute the gating signal, or approximate it with a cheap lex/embedding pre-probe of its own corpus (high top-hit similarity ⇒ relevant fact exists ⇒ inject; low ⇒ skip). It turns 'recall now?' into a threshold, not a prompt rule. Catch: TARG assumes you control the drafting model's logits; qmemd doesn't control Claude, so it must run its own probe model rather than reading Claude's confidence — extra latency on every turn that must stay sub-100ms to be worth it.
- **Sources:** https://arxiv.org/pdf/2511.09803; https://arxiv.org/html/2501.12835v2; https://aclanthology.org/2025.acl-long.319/

#### claude-mem / SuperBrain — hook-based PUSH injection (and the v3 context-pollution cautionary tale)  ·  [W1, W2, W3]  ·  🟢 production

- **What:** Memory plugins that inject relevant prior context via Claude Code lifecycle hooks (SessionStart, UserPromptSubmit) instead of an MCP tool the agent must remember to call — 'capture and inject without Claude having to choose to.'
- **Mechanism:** Hooks write to hookSpecificOutput.additionalContext, which Claude Code silently folds into the context window. SuperBrain runs a SessionStart hook that does project-scoped retrieval and injects the relevant journal slice before the user's first prompt; capture hooks (PostToolUse/Stop/SessionEnd) append NDJSON and a detached child process distills it — hooks always exit 0, never block the turn. CRITICAL caution: claude-mem v3 auto-injected everything at session start, hit 'context pollution — thousands of tokens of irrelevant data,' and v4 RETREATED to an ~800-token index that again requires an explicit search. So unconditional push fails; the winning shape is relevance-GATED push.
- **Relevance to qmemd:** This is the highest-fit transplant: qmemd already owns the exact hook surface (its beacon is a PreToolUse hook; its snapshot runs at session start) — it just uses them to NUDGE, not to INJECT. Moving the beacon from 'nudge the agent to call recall' to 'run a gated recall and inject the top-k as additionalContext on UserPromptSubmit' converts pull→push and removes W1/W2 outright, and deletes the long recall rules (W3) because the machine enforces it. The catch is precisely claude-mem's v3 scar: qmemd MUST gate injection by its existing relevance floor (belowFloor) and project scope, or it recreates the context-pollution failure. qmemd's 'partial snapshot' design is actually the right instinct already; the upgrade is per-prompt gated injection, not bigger session-start dumps.
- **Sources:** https://docs.claude-mem.ai/hooks-architecture; https://alexandrekhoury.com/writing/superbrain-session-memory-claude-code; https://code.claude.com/docs/en/hooks

#### Mem0 — LLM-arbitrated ADD / UPDATE / DELETE / NOOP consolidation  ·  [W5, W4]  ·  🟢 production

- **What:** On every new candidate fact, Mem0 retrieves the top-s semantically similar existing memories and hands them + the candidate to an LLM via a function-calling 'tool call' that must choose exactly one of ADD, UPDATE, DELETE, NOOP.
- **Mechanism:** The LLM resolves conflicts itself by semantic relationship: novel ⇒ ADD; complementary ⇒ UPDATE (merge); contradicted ⇒ DELETE the stale memory; redundant/irrelevant ⇒ NOOP. This is exactly the auto-resolution qmemd deliberately refuses — Mem0 silently overwrites/merges so the store stays coherent without human review. (Note: some accounts describe the OSS single-pass path as ADD-heavy; the paper's full pipeline specifies all four ops.)
- **Relevance to qmemd:** Direct answer to W5 (and W4, since DELETE prunes the stale side of a contradiction). qmemd's Tier-2.5 already DETECTS the contradiction (differing identifier / polarity / antonym) and surfaces it — it stops one step short of Mem0's resolution. qmemd could add an opt-in 'resolve' that feeds the detected conflict + authority comparison to an LLM and applies UPDATE/DELETE. Catch: Mem0 spends an LLM call per write (latency + nondeterminism) and a wrong DELETE silently destroys a true fact — which is exactly why qmemd surfaces instead. The reconciliation: qmemd's git-commit-before-reindex + doctor .bak give it an undo Mem0 lacks, so an LLM-proposed (not silent) UPDATE/DELETE with git as the safety net is the safe middle path.
- **Sources:** https://arxiv.org/html/2504.19413v1; https://github.com/mem0ai/mem0; https://mem0.ai/research

#### Zep / Graphiti — bi-temporal edge invalidation (auto-supersession + auto-staleness)  ·  [W4, W5]  ·  🟢 production

- **What:** A temporal knowledge graph where every fact (edge) carries four timestamps: created_at / expired_at (system/provenance time) and valid_at / invalid_at (when it was true in the world). New contradicting facts don't overwrite — they invalidate.
- **Mechanism:** When an incoming fact contradicts an existing edge, Graphiti closes the old edge's validity window (sets invalid_at) and opens a new edge — superseded, not deleted, with full queryable history. Queries return only the currently-valid slice by default, so stale facts auto-drop out of retrieval without a manual review tax, while the audit trail survives. Staleness and supersession are properties of the data model, not chores.
- **Relevance to qmemd:** qmemd ALREADY has the frontmatter bones for this — supersedes / superseded_by / conflicts_with / updated fields exist — but nothing sets them automatically; review_by/staleness is manual (W4) and conflicts are surface-only (W5). Borrow the invalidation rule: when Tier-2.5 finds a newer fact contradicting an older one from a lower-authority source, auto-stamp superseded_by/superseded-on (closing the old edge) and let recall filter superseded facts out of the default slice — auto-staleness with zero manual 'reviewed' calls, history preserved in git. Catch: Graphiti's power comes from a graph store + entity/relationship extraction qmemd doesn't have; qmemd would get a flat-file approximation (per-fact supersession, no relational temporal queries), and 'which fact is newer/higher-authority' is a heuristic that can mis-invalidate — so default to filter-don't-delete (mirroring Zep's invalidate-don't-delete).
- **Sources:** https://arxiv.org/abs/2501.13956; https://github.com/getzep/graphiti; https://blog.getzep.com/beyond-static-knowledge-graphs/

#### Letta sleep-time agents (background memory consolidation between turns)  ·  [W4, W5, W3]  ·  🟢 production

- **What:** A second 'sleep-time' agent shares the primary agent's memory blocks and runs asynchronously in the background / during idle, rewriting memory rather than serving the user turn.
- **Mechanism:** On a background heartbeat (no user input), the sleep-time agent consolidates fragmented memories into coherent entries, deduplicates and reorganizes blocks, rewrites blocks that have grown messy/bloated, summarizes recent conversation into stable notes, and archives/prunes outdated items. Because it runs off the hot path, dedup/staleness/consolidation cost zero main-turn latency — the maintenance tax is moved to idle time and automated.
- **Relevance to qmemd:** Addresses W4 (auto-prune stale), W5 (auto-dedup/consolidate near-duplicates), and even W3 (it compresses/rewrites bloated memory, which is the storage analog of rule bloat). qmemd already ships a warm daemon (per recent commits) — that daemon is the natural host for a periodic 'sleep-time' sweep that runs reindex + stale detection + Tier-2.5 dedup and PROPOSES merges/supersessions. Catch: Letta's agent has autonomous write access and an LLM loop; qmemd's daemon is currently mechanical (lex/embed, no generative reasoning). Giving it silent write authority reintroduces the wrong-merge/wrong-delete risk qmemd avoids — so scope the sweep to git-reversible proposals (or auto-apply only the mechanical subset doctor already fixes), keeping generative consolidation opt-in.
- **Sources:** https://docs.letta.com/guides/agents/architectures/sleeptime/; https://www.letta.com/blog/sleep-time-compute/; https://forum.letta.com/t/sleeptime-agents-for-memory-consolidation-best-practices-guide/154

**Key takeaways:**

- The field's verdict on W1/W2: stop trusting agent discipline; derive a trigger the agent can't forget. Two implementable families exist for qmemd — (a) GATED PUSH via hooks (claude-mem/SuperBrain inject memory through SessionStart/UserPromptSubmit additionalContext, no tool call), and (b) a cheap WHEN-TO-RETRIEVE GATE (Adaptive-RAG classifier / TARG training-free uncertainty threshold). qmemd already owns the hook surface (beacon=PreToolUse, snapshot=session-start) but uses it only to NUDGE — the single highest-leverage change is converting that nudge into a relevance-gated recall-and-inject.
- Self-RAG and FLARE are the most elegant 'machine decides' answers but are architecturally OUT of reach for qmemd: both require the generator's own tokens/logprobs (fine-tuned reflection tokens, or token-probability thresholds). qmemd sits behind a closed Claude model via MCP and sees neither. Anything qmemd builds must run its OWN small local signal model (it already loads embeddinggemma-300M) or a corpus pre-probe — it cannot reach inside Claude's decoding.
- Heed claude-mem's v3→v4 scar before auto-injecting: v3 pushed all memory at session start, caused 'context pollution (thousands of tokens of irrelevant data),' and v4 retreated to an ~800-token index needing explicit search. This VALIDATES qmemd's 'partial snapshot' instinct and proves the winning shape is GATED push, not always-push — qmemd must reuse its existing relevance floor (belowFloor) + project scope as the injection gate.
- Always-retrieve is a hard-to-beat but expensive baseline: TARG and the ACL 2025 survey of 35 methods show 'Always-RAG' pays a latency/token tax and can even underperform 'Never-RAG' when hits are noisy, while a cheap uncertainty gate matches accuracy at 0.8-30% retrieval rates. So qmemd should NOT swing from 'agent must remember' to 'inject everything every turn' — a lightweight gate beats both extremes.
- On W3 (rule bloat): the cure is not better prose. Every paragraph added to the recall rules is an admission that the trigger lives in the wrong layer. Adaptive-RAG/TARG/hooks all MOVE the trigger out of the prompt into a deterministic external component the machine enforces — porting the trigger to a hook/gate lets qmemd DELETE recall paragraphs rather than add them.
- On W4/W5: production memory systems have converged on auto-resolution — Mem0 (LLM picks ADD/UPDATE/DELETE/NOOP per write), Graphiti (bi-temporal edge invalidation closes the stale fact's validity window), Letta (background sleep-time dedup/prune/consolidate). qmemd's 'surface, never auto-resolve' is the conservative outlier. It already has the detection (Tier-2.5) and the data fields (supersedes/superseded_by/review_by) — the gap is the resolution step. The safe upgrade unique to qmemd: every write is git-committed and doctor writes .bak, so qmemd can offer LLM-PROPOSED, git-reversible supersession/merge (ideally on the warm daemon as a Letta-style sleep-time sweep) without the silent-wrong-delete risk that justifies its current caution.
- Proactive-retrieval-from-MEMORY (not just doc-RAG) is now its own research line — ProactAgent (2026) learns 'when and what to retrieve' from a typed experience base (factual/episodic/skill memory) via RL with paired with/without-retrieval reward, cutting retrieval overhead. It mirrors qmemd's exact domain (typed durable memory) and signals where the puck is going: retrieval timing learned from whether a memory hit actually improved the outcome — implying qmemd should start LOGGING recall-hit usefulness now, since every adaptive gate (trained or training-free-calibrated) needs that signal.

### Angle: prompt-bloat

_prompt-bloat: keeping the agent instruction/rule surface small while preserving reliable recall behavior. Survey of how mature systems get reliable behavior from SHORT instructions — progressive disclosure, tool-description-as-prompt, harness-side injection/system-reminders, adaptive-retrieval gating, recitation/structural enforcement — plus the auto-decay/auto-merge mechanisms that would let qmemd drop maintenance-tax prose (W4/W5)._

#### Anthropic Agent Skills — three-level progressive disclosure  ·  [W3, W2]  ·  🟢 production

- **What:** A skill is a directory whose SKILL.md carries YAML name+description (the only thing pre-loaded into the system prompt) plus a body and linked reference files that load on demand. Claude knows a skill exists and when to use it, but pays zero context for the body until a task matches.
- **Mechanism:** Level 1 = name+description always in system prompt (~tens of tokens). Level 2 = SKILL.md body, read via Bash only when the description matches the task. Level 3+ = bundled reference/*.md and scripts the body links to, navigated only as needed; script CODE never enters context, only stdout. Anthropic's explicit guidance: keep SKILL.md under ~5k words, move detail to references/, and for critical/deterministic checks bundle a SCRIPT rather than prose because 'code is deterministic; language interpretation isn't.' The irreducible always-loaded surface is just the description/trigger sentence.
- **Relevance to qmemd:** qmemd ALREADY ships a qmemd-memory skill, yet the multi-paragraph recall rules also live inline in qmemd.md (always-loaded). The fix: collapse qmemd.md to a one-line pointer + the irreducible TRIGGER LIST, and move the long 'how recall scoping/snapshot/footer works' prose into the skill body / a references/recall.md that loads only when the agent is actually doing memory work. Catch: progressive disclosure is model-INVOKED — the skill body loads only if the model decides the description matches, which re-introduces the discipline problem (W2) for the rule text itself, and the always-loaded trigger sentence is exactly the surface you can't shrink. So invest your few hundred always-on tokens in the WHEN (triggers), not the HOW.
- **Sources:** https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills; https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview; https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf; https://code.claude.com/docs/en/agent-sdk/skills.md

#### Tool-description-as-prompt — put the 'when to use' in the schema, not the system prompt  ·  [W3, W2, W1]  ·  🟢 production

- **What:** Anthropic and every practitioner guide agree the tool DESCRIPTION is the single highest-leverage factor in tool-use reliability: be prescriptive about WHEN to call it (positive AND negative triggers), not just what it does. The description absorbs the prompt-engineering you'd otherwise put in the system prompt.
- **Mechanism:** Description structure: sentence 1 what it does; sentence 2 when to use (trigger conditions, 'Call this when the user asks about X'); sentence 3 when NOT to use (the 'when-not-to' exclusion list does real work preventing misfires); sentence 4 what it returns. Claude Code's own data-tool-use notes: 'On recent Opus models, which reach for tools more conservatively, trigger conditions in the description give measurable lift in should-call rate.' Aim 3-4+ sentences; enums + per-parameter descriptions + input_examples for format-sensitive args. This moves recall's trigger logic from prose nagging into the MCP recall tool's own contract.
- **Relevance to qmemd:** qmemd's recall MCP tool description should carry the full trigger list verbatim ('before diagnosing any build/env error; on first touch of a repo/tool this session; when the user references past context; when an instruction names a mechanism you're about to implement') plus the negative case — deleting those paragraphs from qmemd.md. CATCH (sharp, qmemd-specific): the mcp-data-server issue #14 documents that Claude Code DEFERS MCP tool schemas — only tool NAMES are visible until ToolSearch fetches the schema, so an agent that plans/reasons without first calling recall never sees the guidance at all. Tool-description guidance is reliable only in clients that eagerly load schemas (VS Code Copilot); in deferred-schema clients it has the SAME blind spot as the system prompt for the 'should I even retrieve?' decision. Mitigation they landed on: echo a one-line 'fetch the recall schema before X' pointer in an output the agent reliably reads at session start.
- **Sources:** https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools; https://github.com/Piebald-AI/claude-code-system-prompts/blob/HEAD/system-prompts/data-tool-use-concepts.md; https://github.com/boettiger-lab/mcp-data-server/issues/14; https://kindatechnical.com/claude-ai/defining-tool-schemas-input-schema-and-best-practices.html; https://menuagentic.com/field-guide/tool-use/

#### Harness-side event-driven injection (PreToolUse/UserPromptSubmit hooks + system-reminders)  ·  [W1, W2]  ·  🟢 production

- **What:** The 'put the rules AROUND the agent, not IN it' pattern: deterministic lifecycle hooks fire on conversation events and inject text via additionalContext, wrapped by the harness in a <system-reminder>. This is the mechanism that lets you STOP relying on the model to remember a rule — execution is guaranteed, not model-discretionary.
- **Mechanism:** Claude Code hook cadences: per-session (SessionStart/End), per-turn (UserPromptSubmit/Stop), per-tool-call (PreToolUse/PostToolUse). additionalContext is injected at the matching point as a system-reminder Claude reads on the next request (CLAUDE.md=read once attention varies; Skills=may be skipped; Hooks=guaranteed every matching call). System-reminders are reactive (fire on a predicate over the event log — e.g. 'last 3 bash results all errors', 'context growing large', 'file modified externally'), injected silently, and the model is pre-told via the system prompt to trust <system-reminder> tags. Key authoring rule: write FACTUAL statements ('This repo uses bun test'), not imperative commands, or you trip the model's prompt-injection defenses and it surfaces the text to the user instead of acting on it.
- **Relevance to qmemd:** qmemd ALREADY owns this exact surface — SessionStart hook injects the partial snapshot, PreToolUse 'qmemd hook beacon' injects the tag histogram, Stop injects the write-beacon — but all only NUDGE. The upgrade that directly kills W1/W2: have the PreToolUse/UserPromptSubmit hook actually RUN a cheap lex recall keyed on cwd basename + the user prompt and inject the top-k FACTS as additionalContext, turning the nudge into an injection so the machine performs the retrieval the model keeps forgetting to request. Catches: (1) injecting on every Bash call pollutes context and breaks KV-cache prefix stability (Manus) — must stay gated/throttled like the existing beacon (repo-pivot + every-N); (2) PreToolUse additionalContext was silently dropped for a stretch (issue #19432, fixed only in v2.1.9) — client-version-fragile, so keep a systemMessage fallback; (3) phrase injected facts as statements, not 'you MUST recall', to avoid injection-defense backfire.
- **Sources:** https://code.claude.com/docs/en/hooks.md; https://dev.to/sasha_podles/claude-code-using-hooks-for-guaranteed-context-injection-2jg; https://michaellivs.com/blog/system-reminders-steering-agents/; https://codeaholicguy.com/2026/06/17/i-read-every-claude-code-hook-so-you-dont-have-to/; https://github.com/anthropics/claude-code/issues/19432

#### Adaptive-retrieval gating — decide WHETHER/WHEN to retrieve (Self-RAG, PASK IntentFlow, MemCog proactive protocol)  ·  [W1, W2]  ·  🔬 research

- **What:** A learned or rule policy decides retrieve-now instead of trusting the agent to remember to call recall. Replaces discipline with a cheap binary gate, and only retrieves when it actually helps (skipping retrieval for 2+2-type turns).
- **Mechanism:** Self-RAG trains a Retrieve∈{yes,no,continue} reflection token with a TUNABLE probability threshold (raise threshold→retrieve less; on PopQA skipping retrieval cost 40% accuracy, on PubHealth only 2% — so the gate is task-sensitive). PASK trains a streaming IntentFlow classifier making a DEMAND/NO_DEMAND decision on every turn from the message + recent history. MemCog's Proactive Reasoning Protocol is a SYSTEM-PROMPT spec (not a model) enumerating exact trigger conditions: (i) mentions of people/places/topics that link to stored memory, (ii) the conversation would benefit from historical context, (iii) a user statement CONTRADICTS stored info. Generative-Agents-style importance/recency/relevance scoring is the retrieval-side analogue.
- **Relevance to qmemd:** qmemd already ships embeddinggemma-300M locally and already has a hand-built gate (the beacon's first-touch latch + repo-pivot + every-N throttle). The borrowable move: feed cwd-basename + user prompt through a cheap local gate (even a keyword/embedding heuristic, no training) to decide whether the hook should auto-inject recalled facts, formalizing MemCog's three trigger conditions. CATCH (the load-bearing finding from PASK): most models/gates are good at EITHER always-firing OR never-firing but not both — a gate tuned too hot spams irrelevant facts (context pollution, the very W3 problem), too cold misses the moment memory would help. You need a precision/recall-tuned threshold, and qmemd has no labeled trigger data; the pragmatic substitute is exactly the deterministic first-touch/pivot heuristic the beacon already encodes — upgrade it to inject, don't replace it with an untrained classifier.
- **Sources:** https://arxiv.org/abs/2310.11511; https://github.com/akariasai/self-rag; https://mem0.ai/blog/proactive-memory-in-ai-agents-a-developer-s-guide; https://ar5iv.labs.arxiv.org/html/2605.28046

#### Mem0 auto-merge — LLM-decided ADD/UPDATE/DELETE/NONE with 'latest truth wins'  ·  [W5]  ·  🟢 production

- **What:** On every write, an LLM compares each incoming fact against semantically-retrieved existing memories and emits one of four operations, reusing the original memory ID so contradictions OVERWRITE rather than accumulate. This is precisely the auto-resolution qmemd deliberately refuses to do (W5).
- **Mechanism:** Custom UPDATE_MEMORY_PROMPT carries a decision table: ADD (new fact), UPDATE (same topic, different/richer info — keep ID, keep most-informative text), DELETE (contradiction or explicit removal), NONE (already present/irrelevant). Hash-based (MD5) dedup catches exact duplicates pre-LLM. 'Latest truth wins' on contradiction.
- **Relevance to qmemd:** qmemd could turn its Tier-2.5 contradiction classifier output into a SUGGESTED disposition with the 4-action verb + old/new text, instead of just surfacing a conflict. But the mature-system signal is a CAUTION, not an endorsement: (a) it puts an LLM in the write path — cost + nondeterminism, which violates qmemd's 'writes are lex-only, no model' invariant; (b) Mem0's own PR #4903 documents the UPDATE prompt HALLUCINATING the old string back into the text field, discarding the new truth; (c) critically, Mem0 v3 RETREATED from UPDATE/DELETE auto-merge to ADD-only + linked_memory_ids precisely because explicit auto-resolution was too lossy/risky — 'both memories being stored is intentional behavior.' So the leading library partially ABANDONED auto-resolve, which validates qmemd's surface-don't-resolve stance. Borrow only the structured 4-action decision table as a human-confirmed suggestion, never silent application.
- **Sources:** https://github.com/mem0ai/mem0/blob/v1.0.10/docs/open-source/features/custom-update-memory-prompt.mdx; https://github.com/mem0ai/mem0/pull/4903; https://github.com/mem0ai/mem0/blob/main/mem0-plugin/skills/mem0/references/architecture.md; https://docs.mem0.ai/core-concepts/memory-operations/add

#### Zep/Graphiti bi-temporal invalidation — invalidate-not-delete, query the 'currently valid' slice  ·  [W4, W5]  ·  🟢 production

- **What:** Every fact/edge carries validity windows (valid_at, invalid_at) plus transaction times (created_at, expired_at). When a new fact contradicts an old one, the system sets the old edge's invalid_at/expired_at instead of deleting it; queries default to the currently-valid slice. Staleness becomes structural — old facts drop out of 'what's true now' with zero manual review tax, but stay auditable.
- **Mechanism:** On ingest, an LLM compares each new edge against semantically-related existing edges (edge_operations.py) for temporal contradiction; on a temporally-overlapping conflict it sets old.invalid_at = new.valid_at and old.expired_at = now, sorting by valid_at to decide which to expire. History is preserved (non-lossy), so point-in-time queries ('what was true on date X', 'what's true now') and provenance-to-source-episode both work. OpenAI's Temporal Agents cookbook adds an explicit invalidated_by link.
- **Relevance to qmemd:** qmemd already has supersedes/superseded_by frontmatter — adopting invalidate-not-delete + a default 'current slice' recall filter would auto-suppress stale facts WITHOUT the manual 'reviewed' command, while keeping the file for audit. Catches: (1) invalidation is driven by an LLM contradiction classifier at WRITE time — the model-in-write-path qmemd avoids; (2) it only fires on EXPLICIT contradiction — a fact nobody contradicts but that is simply old (pure decay) is never invalidated, so this solves superseded-by-contradiction (a W5-flavored staleness) but NOT decay-without-contradiction (true W4), which still needs qmemd's review_by time policy; (3) bi-temporal value (point-in-time queries) is overkill for durable project facts where you almost always want 'now'.
- **Sources:** https://arxiv.org/html/2501.13956v1; https://getzep-graphiti.mintlify.app/concepts/temporal-model; https://github.com/getzep/graphiti; https://blog.getzep.com/beyond-static-knowledge-graphs/; https://developers.openai.com/cookbook/examples/partners/temporal_agents_with_knowledge_graphs/temporal_agents

#### Ebbinghaus forgetting curve — recall-strengthened decay (MemoryBank, FadeMem)  ·  [W4]  ·  🔬 research

- **What:** Model each memory's retention as exponential decay R = e^(−t/S), where S is strength; usage IS the review signal — every time a memory is RECALLED, S increments and t resets (the spacing effect), so frequently-recalled facts persist and never-touched ones fade. Makes freshness self-maintaining with zero manual review command.
- **Mechanism:** MemoryBank: S initialized to 1 on first mention; on recall, S += 1 and t := 0, lowering future forget probability; items with low S and large elapsed t are forgotten. FadeMem generalizes to a dual-layer hierarchy with DIFFERENTIAL decay rates modulated by semantic relevance + access frequency + temporal pattern, plus LLM-guided fusion to consolidate redundant items — large storage savings with better multi-hop retrieval on LoCoMo/MSC.
- **Relevance to qmemd:** The cheap, safe half to steal: make RECALL automatically reset the review clock (forward-set review_by / bump an access-count strength field) — qmemd's 'reviewed' command does exactly this but MANUALLY; wiring it into recallQuery converts usage into automatic maintenance and erases the W4 tax for any fact that's actually being used. CATCH (decisive): pure time-decay + auto-deletion is DANGEROUS for qmemd's durable facts — a critical-but-rarely-recalled gotcha (a once-a-year build fix, a destructive-command warning) would decay precisely when you finally need it. qmemd's per-type durable defaults (user/feedback never expire; project 90d/reference 180d) already encode this objection. Borrow 'recall strengthens / resets the clock'; do NOT borrow auto-forget for project/reference truth.
- **Sources:** https://ojs.aaai.org/index.php/AAAI/article/view/29946; https://github.com/zhongwanjun/MemoryBank-SiliconFriend; https://arxiv.org/pdf/2601.18642; https://export.arxiv.org/pdf/2305.10250v3.pdf

#### Recitation & structural enforcement — make the behavior the harness's job, not the prompt's (Manus todo.md, Anthropic harness feature-list)  ·  [W2, W3]  ·  🟢 production

- **What:** Instead of nagging via long prompt rules, encode the desired behavior as a STRUCTURE the harness maintains/enforces, so the model physically cannot skip it. Two flavors: continuous goal recitation (rewrite a todo.md into the context tail every step to exploit recency bias) and hard structural gates (commit-discipline + inviolable tests + a feature-list file).
- **Mechanism:** Manus: agent rewrites todo.md each step, reciting objectives into the high-attention recency zone to beat lost-in-the-middle and goal drift over ~50-tool-call loops — natural language biasing its own attention, no architecture change. Anthropic harness engineering: an init.sh + claude-progress.txt + JSON feature-list + git commits mean 'the model cannot skip incremental progress (commit discipline), cannot declare premature victory (inviolable tests), cannot lose the plan (feature list) — the harness enforces them structurally. The model does not need to remember these practices.' AgentPatterns taxonomy: recitation (agent, every step) vs critical-instruction repetition (author, static, duplicate rules at start+end of prompt) vs event-driven system reminders (harness, reactive).
- **Relevance to qmemd:** Direct answer to 'machine cannot force recall, can only nag': replace the recall-rule paragraphs with a STRUCTURAL artifact — e.g., the SessionStart hook seeds a short 'memory checklist' line ('□ recalled repo facts before diagnosing') that the agent rewrites/checks off, keeping the trigger salient in the recency zone instead of buried in a static rule block; or make the first build-error a structural gate that the hook intercepts. This also shrinks rule surface (W3): a 6-word recited checklist item beats three paragraphs. Catches: (1) recitation costs ACTIONS — Manus found the executor wasted ~1/3 of its actions on todo bookkeeping and had to move it to a dedicated planner sub-agent; (2) recitation is EPHEMERAL — lost on compaction unless explicitly preserved; (3) recitation keeps the GOAL salient but does NOT itself fetch memory — you still need the injection (finding 3) or gate (finding 4) to actually retrieve.
- **Sources:** https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus; https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents; https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents; https://agentpatterns.ai/context-engineering/goal-recitation/

**Key takeaways:**

- The dominant pattern in mature systems for W1/W2 (auto-trigger + discipline) is 'put the behavior AROUND the agent, not in the prompt': deterministic harness hooks inject context or enforce structure so correctness no longer depends on the model remembering. qmemd already owns the entire hook surface (SessionStart snapshot, PreToolUse beacon, Stop write-beacon) — the single highest-leverage change is to make the gated PreToolUse/UserPromptSubmit hook actually RUN a cheap lex recall and INJECT the top-k facts as additionalContext, converting today's nudge into an injection. That is the smallest change that turns 'agent must remember to recall' into 'the machine retrieves for it.'
- For W3 (rule-surface bloat) the canonical fixes are progressive disclosure (Skills) and tool-description-as-prompt — but BOTH have the same deferred-loading failure mode in Claude Code: a Skill body loads only if the model decides to invoke it, and an MCP tool's schema/description isn't even visible until ToolSearch fetches it (mcp-data-server #14). The irreducible always-loaded surface is therefore the TRIGGER LIST. Concrete recommendation: shrink qmemd.md to a one-line pointer + the bare WHEN-to-recall triggers, move all the HOW (scoping, snapshot, footer semantics) into the qmemd-memory skill body / references, and duplicate the triggers into the recall tool description.
- Adaptive-retrieval gating (Self-RAG threshold, PASK IntentFlow per-turn classifier, MemCog's three explicit trigger conditions) shows the central tension: an auto-trigger that fires too often pollutes context (recreating the bloat problem) while one that fires too rarely misses — 'most gates are good at always-firing OR never-firing, not both.' qmemd's existing first-touch latch + repo-pivot + every-N throttle is already a sensible hand-built gate; upgrade it to inject rather than replace it with an untrained classifier it has no labeled data to fit.
- For W4/W5 (staleness, dedup/conflict) the field converged on 'invalidate/decay, don't delete' (Zep bi-temporal, MemoryBank/FadeMem Ebbinghaus) and 'latest truth wins' (Mem0) — BUT the leading library, Mem0 v3, retreated from LLM auto-merge back to ADD-only because write-path resolution proved lossy and hallucination-prone (PR #4903). That validates qmemd's deliberate 'surface, never auto-resolve' stance.
- The two cheapest, safest borrows that remove qmemd's manual maintenance tax without an LLM in the write path: (a) make recallQuery auto-reset review_by on access (the Ebbinghaus spacing effect — usage becomes the review signal, so qmemd's manual 'reviewed' command becomes automatic for any fact actually in use); and (b) add a default 'currently-valid slice' recall filter over the existing supersedes/superseded_by frontmatter (Zep-style invalidate-not-delete) so superseded facts stop surfacing without being destroyed. Both are time/metadata operations, not model calls — they fit qmemd's lex-only-write invariant.
- Recitation (Manus todo.md) and structural enforcement (Anthropic feature-list + inviolable tests) are the prose-replacement answer to W2/W3: a 6-word recited checklist item in the recency zone outperforms three static rule paragraphs at keeping a trigger salient. But recitation only keeps the GOAL salient — it doesn't fetch memory and it costs actions (Manus moved todo upkeep to a dedicated sub-agent after it ate ~1/3 of executor actions), so it composes with, rather than replaces, hook-injection and gating.

### Angle: lifecycle-decay-conflict

#### mem0 ADD/UPDATE/DELETE/NOOP resolver — and its v3 reversal to ADD-only  ·  [W5]  ·  🟢 production

- **What:** mem0's original 'update phase' auto-resolves each newly-extracted fact against existing memory by having an LLM pick one of four ops. In v3 (2026) mem0 deliberately reverted to ADD-only and stopped auto-deleting/updating.
- **Mechanism:** After extraction, for each candidate fact the system retrieves top-s semantically-similar memories and hands {candidate, neighbors} to the LLM via a function-calling 'tool call'. The LLM itself selects ADD (no equivalent exists), UPDATE (augment), DELETE (existing memory contradicted), or NOOP — no separate classifier; the LLM's reasoning IS the resolver. CATCH: v3 abandoned this. New pipeline is single-pass ADD-only + MD5 hash dedup; 'when information changes, both old and new survive' and ranking surfaces the current one. mem0's own docs attribute +42.1 temporal and +23.1 multi-hop gains to ADD-only, saying the UPDATE/DELETE model 'would destroy' chronological context. The field's most-cited auto-resolver was measured to lose information and was rolled back.
- **Relevance to qmemd:** Strongest validation of qmemd's 'surface, never resolve' instinct: canonical auto-DELETE destroyed temporal context and was reverted. Borrowable middle path is v3's 'accumulate + rank current-first' — don't block/merge near-dups, keep both, let recall ranking float the live one. qmemd already has supersedes/superseded_by; it could auto-set supersedes (soft, reversible, git-audited) rather than blocking or overwriting. Catch: accumulation needs a recency/supersession tiebreaker in lex/hybrid ranking or the stale dup resurfaces.
- **Sources:** https://arxiv.org/html/2504.19413v1; https://docs.mem0.ai/core-concepts/memory-evaluation; https://docs.mem0.ai/migration/oss-v2-to-v3; https://github.com/mem0ai/mem0

#### Zep/Graphiti bi-temporal edge invalidation (non-lossy auto-resolution)  ·  [W5, W4]  ·  🟢 production

- **What:** Graphiti auto-resolves contradictions WITHOUT deleting: it invalidates the superseded fact via temporal metadata and keeps it as history, so 'current state' and 'what was true at time T' are both queryable.
- **Mechanism:** Every edge carries four timestamps: created_at / expired_at (transaction time: when the system learned/unlearned) and valid_at / invalid_at (valid time: when the fact was true in the world). On each new episode an LLM compares the new edge against semantically-related existing edges (edge_operations.py) to detect contradiction. On a temporally-overlapping contradiction it sets the old edge's invalid_at = new fact's valid_at and expired_at = now, then creates a new edge — new info always wins on the transaction timeline, old edge preserved. Retrieval of 'current' filters on expired_at/invalid_at null. This is automatic conflict RESOLUTION that is still non-lossy.
- **Relevance to qmemd:** Shows you can auto-resolve and keep history simultaneously — the resolution is a state transition (set invalid_at), not a delete. qmemd's frontmatter already has supersedes/superseded_by/conflicts_with plus created/updated; adding valid_at/invalid_at (or repurposing review_by as invalid_at) would let qmemd auto-mark the older of a contradicting pair superseded and exclude it from default recall while keeping the file. Also doubles as staleness (W4): a fact past invalid_at is auto-dropped from current recall without manual 'reviewed'. Catch: requires a per-write LLM contradiction check over neighbors (qmemd's Tier-2.5 is model-free by design) and recall must learn to filter on the temporal fields; full graph not needed, flat files + frontmatter suffice.
- **Sources:** https://getzep-graphiti.mintlify.app/concepts/temporal-model; https://arxiv.org/html/2501.13956v1; https://github.com/getzep/graphiti; https://www.getzep.com/platform/graphiti/

#### Generative Agents retrieval scoring (recency·importance·relevance) + threshold-fired reflection  ·  [W4, W1]  ·  🔬 research

- **What:** The seminal memory-stream design auto-retrieves by a composite score with no agent decision, and auto-consolidates via reflection that fires on accumulated importance rather than a schedule or manual command.
- **Mechanism:** Each memory is scored score = a·recency + b·importance + c·relevance, all min-max normalized to [0,1]. Recency = exponential decay (0.995^hours-since-last-access); importance = an LLM-assigned 'poignancy' rating (1=idle .. 10=critical) stored at write time; relevance = cosine to the focal-point query. Reference weights are relevance 3, importance 2, recency 0.5; top-30 enter context. CONSOLIDATION TRIGGER: the running SUM of poignancy scores drives reflection — when cumulative importance crosses a threshold, the LLM generates salient questions, synthesizes higher-level 'thought' nodes, and writes them back into the stream (compressing many low-level memories into fewer retrievable insights).
- **Relevance to qmemd:** Two borrowable ideas. (1) Write-time importance/poignancy as a frontmatter field gives qmemd a salience signal to rank by and to gate writes (low-poignancy facts need not persist) — softer than a binary remember. (2) A decay term in recall ranking auto-deprioritizes stale facts WITHOUT deleting them, replacing the manual review_by tax with continuous soft-decay. (3) The threshold-fired reflection is a model for auto-consolidation: when N new project facts accumulate, run a synthesis pass instead of waiting for a human. Catch: pure recency decay buries durable user/feedback facts — qmemd must exempt those types (it already treats user/feedback as durable) and importance scoring costs an LLM call per write.
- **Sources:** https://arxiv.org/pdf/2304.03442; https://agentpatterns.ai/agent-design/generative-agents-memory-stream/; https://dl.acm.org/doi/fullHtml/10.1145/3586183.3606763

#### MemoryBank Ebbinghaus decay + MemoryOS heat-based eviction (fully automatic staleness)  ·  [W4]  ·  🔬 research

- **What:** Two systems make staleness automatic via usage-and-time scoring, with zero manual review: rarely-used memories decay or get evicted; frequently-recalled ones auto-strengthen.
- **Mechanism:** MemoryBank: retention/importance = e^(-Δt/S) (Ebbinghaus), where Δt = time since last recall and S = memory strength = number of times retrieved (init 1). Each recall increments S by 1 and resets Δt=0 — so usage continuously re-strengthens a memory and idleness decays it; weak memories drop below a threshold and are forgotten. MemoryOS: each mid-term segment accumulates a Heat = a·N_visit + b·L_interaction + c·R_recency (retrieval count, engagement length, recency decay); when capacity is exceeded the lowest-heat segments are auto-evicted, and segments whose heat exceeds threshold τ are promoted/distilled into the long-term user profile. Both are closed-loop: recall itself is the maintenance signal.
- **Relevance to qmemd:** Directly attacks qmemd's manual-maintenance-tax weakness. Because qmemd already logs writes and could log recalls, it can compute a MemoryBank-style strength S (recall count) and Δt (since last recall) per fact for FREE (no LLM), then use e^(-Δt/S) as a ranking decay multiplier — facts you actually use stay hot, untouched ones sink. This makes review_by/reviewed largely unnecessary for ranking. Catch: pure eviction is unsafe for a durable knowledge base (a critical fact recalled once a year would be evicted); qmemd should use decay for RANKING/flagging only, never deletion — i.e. heat decides what to surface, not what to keep. Recall-count tracking also adds a write on every read.
- **Sources:** https://arxiv.org/abs/2305.10250; https://arxiv.org/html/2506.06326; https://github.com/BAI-LAB/MemoryOS; https://aclanthology.org/2025.emnlp-main.1318.pdf

#### A-MEM Zettelkasten link-generation + memory evolution (consolidate by refining, not dropping)  ·  [W5, W4]  ·  🔬 research

- **What:** On every new memory, A-MEM auto-builds links to related memories and then triggers an 'evolution' pass that updates the attributes of the NEIGHBORS in light of the newcomer — consolidation that enriches existing facts instead of deleting duplicates.
- **Mechanism:** Three stages per write: (1) Note Construction — LLM generates keywords, tags, and a contextual description for the new note; embed the concatenation. (2) Link Generation — embedding cosine retrieves top-k neighbors; an LLM decides which to link (forming thematic 'boxes'/clusters), no predefined rules. (3) Memory Evolution — for each neighbor m_j, an LLM decides whether the new memory should update m_j's keywords/tags/contextual-description; the updated m_j* replaces the original. So near-duplicates become linked-and-mutually-refined rather than a blocked write or a silent overwrite.
- **Relevance to qmemd:** An alternative to qmemd's conflict-surfacing: instead of stopping at 'disposition:conflict', auto-link the related facts (qmemd has tags already) and optionally run an evolution pass that sharpens the older fact's description/tags using the new one. This consolidates the corpus (fewer redundant facts, richer survivors) — addressing bloat and drift over time. Catch: every write fires multiple LLM calls over neighbors (cost + latency; qmemd's writes are deliberately model-free), and silently MUTATING existing facts risks unreviewed drift — though qmemd's per-write git commit gives an audit/rollback trail that A-MEM lacks, making the mutation safer to adopt.
- **Sources:** https://arxiv.org/html/2502.12110; https://github.com/WujiangXu/A-mem; https://proceedings.neurips.cc/paper_files/paper/2025/file/19909c36f51abc4856b4560aff3d36d6-Paper-Conference.pdf

#### Harness-level proactive memory injection (NeMo auto_memory_agent, MS Agent Framework Context Providers, Letta always-in-context core memory)  ·  [W1, W2, W3]  ·  🟢 production

- **What:** The direct architectural answer to discipline-dependence: a wrapper/middleware retrieves and injects relevant memory BEFORE every agent call. The framework decides to retrieve — the LLM never has to remember to call a recall tool.
- **Mechanism:** NVIDIA NeMo's auto_memory_agent wraps ANY agent with three flags — save_user_messages_to_memory, retrieve_memory_for_every_response (injects relevant context before each agent call), save_ai_messages_to_memory — explicitly motivated by 'LLMs may forget to call memory tools'. Microsoft Agent Framework 'Context Providers' register once and run a before-hook (inject relevant facts/history into the prompt) and after-hook (extract+store new memories) on EVERY invocation; their rule: 'if the agent should have this info every time, use a context provider; if only sometimes, use a tool.' Letta/MemGPT: core memory blocks (and MemFS system/ files) are ALWAYS loaded into the system prompt — durable facts are pinned in-context, not retrieved on demand. Anthropic's memory tool persists files client-side across sessions but is still tool-gated; the injection pattern above is what removes the agent's choice.
- **Relevance to qmemd:** This is the gap qmemd's beacon stops short of: the beacon DETECTS unloaded facts (repo pivot, ~every 20 bash) and nudges, but does not inject — it still leans on the agent to then call recall. Closing W1/W2/W3 means making the existing PreToolUse/SessionStart hook inject the top-k recall results (or pin durable user/feedback/pinned facts as always-in-context, Letta-style) instead of injecting a count and prose. That converts several paragraphs of brittle 'remember to recall' rules (W3) into a deterministic harness behavior. Catch: injection spends tokens on every turn and injects noise if retrieval is weak — so gate it (pin only durable/pinned facts always; inject query-relevant project facts only on detected triggers) and budget it; qmemd's project-scoping already narrows the candidate set.

#### Agentic-RAG cheap router: deterministic trigger classifier instead of long prose rules  ·  [W1, W2, W3]  ·  🟢 production

- **What:** Production agentic-RAG decides whether to retrieve with a fast deterministic router (rules/heuristics first, LLM classifier only for ambiguous cases) rather than a fixed pipeline or an LLM judgment every turn — turning 'when to retrieve' into code, not instructions.
- **Mechanism:** A hybrid router runs deterministic signals first (regex patterns, length heuristics, keyword signals) and escalates to a cheap classifier model (e.g. Haiku) only for ambiguous queries; retrieval is then a bounded loop (cap 3-5 rounds, 'answer with what you have' fallback). Reported distribution: ~90% of queries answer in 1 retrieval, ~8% need 2, ~2% multi-hop. The framing 'retrieval is a tool the agent reasons about, not a pipeline stage the system always executes' is paired with the context-provider rule (always-present info = inject; sometimes-needed = tool) to split memory into auto-injected vs on-demand.
- **Relevance to qmemd:** qmemd's recall-timing rules are several long paragraphs (W3) that the LLM must internalize and obey (W2). A router replaces them with code: qmemd's beacon ALREADY computes the exact deterministic signals these routers use — repo first-touch, repo pivot, and a build/tooling-error context window. Wire those signals to auto-fire a scoped recall (and inject results) rather than to print a nudge. Deterministic trigger > prose rule: every word removed from the rule surface reduces rule fatigue. Catch: false-positive triggers inject irrelevant facts and cost tokens; needs precision tuning and a 'no new facts -> stay silent' guard, exactly like the agentic-RAG min-new-chunks break condition.
- **Sources:** https://datarekha.com/blog/agentic-rag-the-agent-decides/; https://dev.to/aloknecessary/agentic-rag-designing-self-correcting-retrieval-loops-for-production-2lbg; https://learn.microsoft.com/en-us/agent-framework/journey/adding-context-providers

#### Letta sleep-time / MemGPT memory-pressure self-editing (system-triggered async consolidation)  ·  [W4, W5, W1]  ·  🟢 production

- **What:** Memory maintenance is triggered by the SYSTEM, not the user: MemGPT issues 'memory pressure' warnings that prompt the agent to consolidate, and Letta runs idle-time 'sleep-time'/'dream' subagents that dedup, merge, and reorganize memory asynchronously.
- **Mechanism:** MemGPT self-edits core memory via core_memory_append/replace, but the trigger is a system-generated heartbeat/pressure event as context fills — the harness forces the consolidation turn. Letta sleep-time agents run when no user message is pending: a background subagent reviews recent conversation, writes lessons to memory blocks, and a 'memory defragmentation' flow backs up memory then splits large files, merges duplicates, and restructures the hierarchy. A separate 'doctor' subagent validates memory. Crucially these run in git worktrees so they write concurrently without blocking the main agent (MemFS is git-backed).
- **Relevance to qmemd:** qmemd already has the pieces: a doctor (frontmatter audit), stale (review backlog), three-tier dedup, and a git-committed store. Letta shows how to remove the manual tax (W4) and the synchronous-resolve risk (W5): run them as a scheduled/idle 'sleep-time' pass (a cron or post-session hook) that auto-runs doctor --fix, computes the staleness backlog, and merges/supersedes near-dups OFFLINE — where being wrong is cheap and reversible via the git trail, unlike a synchronous block at write time. This also reduces discipline-dependence (W1) by making consolidation a machine routine, not an agent obligation. Catch: needs a scheduler/daemon (qmemd has install-service systemd/launchd scaffolding already) and async auto-merge can still err — keep it to high-confidence merges + supersede (reversible) and leave genuine contradictions surfaced for human review.
- **Sources:** https://www.letta.com/blog/agent-memory/; https://docs.letta.com/letta-code/memory/; https://sureprompts.com/blog/letta-memgpt-walkthrough; https://github.com/letta-ai/letta

**Key takeaways:**

- The field's most-cited auto-resolver (mem0 ADD/UPDATE/DELETE/NOOP) was REVERTED to ADD-only in v3 because auto-DELETE/UPDATE measurably destroyed temporal context (+42.1 temporal, +23.1 multi-hop from the rollback). This is strong evidence FOR qmemd's 'surface, never silently resolve' stance — the safe pattern is accumulate + rank-current-first + soft supersede, not overwrite/delete.
- The only mechanism that truly fixes W1/W2/W3 (discipline-dependence) is moving retrieval OUT of the LLM's hands: harness-level proactive injection (NeMo auto_memory_agent, MS Agent Framework Context Providers) runs a before-every-call hook that injects relevant memory, and Letta pins durable facts always-in-context. The motivating quote across these is literally 'LLMs may forget to call memory tools.' qmemd's beacon already detects the trigger moments but only NUDGES — it should inject.
- Convert qmemd's long prose recall rules (W3) into a deterministic trigger router (agentic-RAG pattern: rules/heuristics first, LLM classifier only if ambiguous). qmemd's beacon already computes the exact signals these routers use (repo first-touch, repo pivot, build-error context) — wire them to auto-fire a scoped recall instead of printing a count.
- For staleness (W4), the borrowable mechanism is usage-driven soft-decay for RANKING, not deletion: MemoryBank's e^(-Δt/S) (S = recall count, auto-incremented on use) and MemoryOS heat scoring make frequently-used facts hot and idle ones sink, with zero manual review — but a durable knowledge base must decay-to-deprioritize, never decay-to-evict.
- Auto-consolidation is best done as a system-triggered or idle-time pass, not synchronously at write: Generative Agents fire reflection on accumulated importance; Letta runs sleep-time/dream subagents (defrag: merge dups, split, reorganize) in git worktrees. qmemd already has doctor + stale + git audit + install-service scaffolding — these could run as a scheduled offline pass where auto-merge is cheap and reversible, instead of a risky synchronous block.
- Non-lossy resolution is a real option (Zep/Graphiti bi-temporal invalidation, A-MEM memory-evolution): contradictions are resolved by a state transition (set invalid_at / supersede / refine neighbor) that keeps the old fact as history. qmemd already has supersedes/superseded_by/conflicts_with frontmatter — it could auto-set supersedes on high-confidence updates while leaving genuine contradictions surfaced, getting most of the auto-resolve benefit without mem0's information loss.
- Recurring cross-system catch: every auto-mechanism that improves on qmemd's manual approach costs either tokens-per-turn (injection), an LLM-call-per-write (importance scoring, contradiction detection, memory evolution), or a background daemon (sleep-time). qmemd's deliberately model-free, write-cheap design means the cheapest wins are: (a) inject what the beacon already knows, (b) usage-count decay (free, no model), (c) offline scheduled doctor/dedup — reserve LLM-in-the-loop resolution for an opt-in async pass.

#### MemGPT — OS-style self-paging: memory-pressure interrupts + self-editing core-memory blocks  ·  [W1, W2, W3]  ·  🟢 production

- **What:** Treats the context window as paged 'main memory' over external storage. The system (not the agent's goodwill) decides when to act on memory via OS-style interrupts, and keeps an always-in-context, self-editable memory block so key facts need no retrieval call.
- **Mechanism:** A queue manager watches token budget against tiered thresholds: at 70% of the window it INJECTS a 'memory pressure' system message prompting the LLM to flush salient content to archival storage; at 100% it evicts ~50% of the FIFO queue and rewrites a recursive summary. 'request_heartbeat=true' lets a function call immediately re-invoke the LLM to chain store/recall steps. 'Core memory' (working context) is a fixed-size in-context text block writeable only via functions (core_memory_append / core_memory_replace) — always present, so persona/prefs are never 'recalled', they just are.
- **Relevance to qmemd:** Two transplantable ideas. (1) The memory-pressure INTERRUPT is the canonical machine-side answer to W1/W2: a deterministic threshold injects a system message instead of trusting the agent to remember a rule. (2) Core-memory blocks attack W3 directly — replace the multi-paragraph recall rules with a tiny always-in-context block holding the few load-bearing facts (current project invariants, active prefs), so the common case needs zero recall call and zero rule-reading. CATCH: MemGPT owns its prompt and control loop; qmemd is an MCP server that does NOT own the host agent's context window or control flow, so it cannot inject system messages or interrupt. The only injection surface qmemd has today is the PreToolUse beacon hook (stdout), which nudges but carries no payload. Adopting either idea requires a harness hook that actually PREPENDS memory content into context (an on-turn hook, not just PreToolUse-Bash) — the decision logic is easy, the injection channel is the missing infrastructure.

#### Generative Agents — retrieval score = recency·importance·relevance + importance-sum reflection cadence  ·  [W1, W4]  ·  🔬 research

- **What:** The reference scoring formula for agent memory retrieval, plus an automatic 'reflect/consolidate' trigger keyed on accumulated importance rather than a clock or manual command.
- **Mechanism:** Every memory is scored score = α_rec·recency + α_imp·importance + α_rel·relevance (Stanford impl sets all α=1), each component min-max normalized to [0,1]. recency = exponential decay γ=0.995 per hour since LAST ACCESS (recall resets it); importance = LLM-rated 1–10 (or heuristic) assigned at write time; relevance = cosine similarity of query vs memory embedding. Reflection fires automatically when the sum of importance scores of recently perceived events exceeds a threshold (150 in the impl, ~2–3×/day): the agent queries the 100 most-recent records, generates higher-level questions, and writes the synthesized answers back as new (retrievable) memories.
- **Relevance to qmemd:** qmemd already has the 'relevance' leg (lex/hybrid score). Adding an importance field (heuristic or one-shot LLM rating, stored in frontmatter alongside created/updated) and a recency term turns its single search score into the principled 3-factor rank, which lets recall self-prioritize what to surface (a soft answer to W1: better defaults reduce dependence on perfect query phrasing). The importance-sum reflection trigger is a concrete W4 pattern: an AUTOMATIC consolidation pass that fires on accumulated salience, synthesizing many raw facts into durable summaries — staleness fought by synthesis, not by a manual review_by clock. CATCH: importance needs either a model call at write (cost) or a heuristic; recency decay needs tracking last-access/recall-count, which qmemd does not store today (it tracks created/updated/review_by). Blanket recency decay is also WRONG for a durable knowledge store — qmemd intentionally keeps user/feedback facts durable; decay would bury stable preferences. Decay must be type-scoped (project/reference only) to stay aligned with qmemd's design.

#### Mem0 — two-phase pipeline with LLM memory-manager ADD/UPDATE/DELETE/NOOP  ·  [W5, W4]  ·  🟢 production

- **What:** The literature's clearest design for AUTO-resolving dedup/conflict: a per-fact LLM decision that picks one of four operations against the existing store, with an append-only audit trail.
- **Mechanism:** Phase 1 (extraction, 1 LLM call): pull candidate facts from the new turn. Phase 2 (update): for each candidate, retrieve the top-k semantically similar existing memories, then hand {candidate + those neighbors} to the LLM, which emits exactly one operation — ADD (genuinely new), UPDATE (augment/refine an existing memory), DELETE (remove a memory the new fact contradicts), or NOOP (already known / irrelevant). Every mutation is logged with old_memory, new_memory, event type, and timestamp (reversible audit trail). A graph variant adds an entity extractor + relationship generator for relational conflict handling. Reported ~90% token/latency savings vs full-context.
- **Relevance to qmemd:** Direct answer to W5. qmemd already does Tier-2.5 contradiction DETECTION (differing identifier / polarity / antonym) and computes an authority comparison, but deliberately stops at 'surfaced conflict' and never auto-resolves. Mem0 supplies the missing decision step (LLM chooses UPDATE vs DELETE vs NOOP) and — crucially — the safety scaffolding qmemd would need to do it responsibly: an append-only event log so any auto-merge is reversible. qmemd's existing --replace/supersedes/superseded_by machinery already IS the UPDATE primitive and a soft-DELETE; bolting the Mem0 operation-picker on top (gated by the authority score qmemd already computes) would let it auto-resolve the obvious cases and only surface the genuinely ambiguous ones, shrinking the W5 maintenance tax. CATCH: Mem0 trusts the LLM to silently DELETE/overwrite — exactly what qmemd's 'engine never auto-resolves' rule guards against; the June-2026 survey documents the failure mode ('after enough passes the agent remembers a sanitized, generic version'). Mitigation: map DELETE→soft supersede (never hard-delete a git-tracked fact), force an audit-log commit per op, and keep a human/agent confirmation for low-authority-gap conflicts. Also adds a model call to every write (qmemd writes are currently lex-only, no model load).

#### A-MEM — Zettelkasten note schema + link generation + memory evolution  ·  [W5, W4]  ·  🔬 research

- **What:** An agentic memory that auto-links new notes to old ones and lets a new note trigger REWRITES of related old notes' metadata — automatic refresh of existing knowledge as a side effect of writing.
- **Mechanism:** Each note mᵢ = {content cᵢ, time tᵢ, keywords Kᵢ, tags Gᵢ, contextual-description Xᵢ, embedding eᵢ, links Lᵢ}; K/G/X are LLM-generated at write (Kᵢ,Gᵢ,Xᵢ ← LLM(cᵢ ∥ tᵢ ∥ prompt)). Link generation: cosine sim sₙ,ⱼ = (eₙ·eⱼ)/(|eₙ||eⱼ|) → take top-k neighbors → LLM decides which to link (Lₙ ← LLM(mₙ ∥ neighbors ∥ prompt)), forming a bidirectional graph. Memory evolution: the new note can update each neighbor's keywords/tags/context-description: mⱼ* ← LLM(mₙ ∥ (neighbors\mⱼ) ∥ mⱼ ∥ prompt), and mⱼ* replaces mⱼ. Mimics human reconsolidation.
- **Relevance to qmemd:** qmemd facts are flat markdown with tags but no inter-fact LINKS and no auto-refresh. A-MEM's note schema maps almost 1:1 onto qmemd frontmatter (name/description/tags already exist; add an LLM 'contextual description' + a links list). Two payoffs: (1) the link graph gives retrieval a structural signal beyond flat tags (better recall defaults → softer W1); (2) memory evolution is a W4 mechanism where adding a new fact AUTOMATICALLY refreshes the wording/tags of related stale facts, no manual review_by. It also overlaps W5: linking surfaces near-duplicates as graph neighbors instead of FTS hits. CATCH: evolution rewrites existing facts on EVERY new write — unbounded model cost and the same silent-mutation/drift risk the survey warns about. For qmemd's git-backed store every evolution is a commit (noise), and links need maintenance on forget() (qmemd already 'reclaims orphans', so the hook exists). Safer adoption: generate links eagerly but gate evolution behind a manual/offline pass rather than on the write path.
- **Sources:** https://arxiv.org/abs/2502.12110; https://arxiv.org/html/2502.12110v1; https://github.com/agiresearch/A-mem

#### MemoryBank — Ebbinghaus forgetting curve with recall-as-reinforcement  ·  [W4]  ·  🔬 research

- **What:** Automatic, access-driven decay: memories fade by an exponential forgetting curve unless recalled, and each recall reinforces them — removing the manual maintenance clock entirely.
- **Mechanism:** Each memory carries a discrete memory-strength S. Retention follows the Ebbinghaus curve R = exp(-t/S) over elapsed time t since last recall. On a successful recall event, S is incremented (spaced-repetition reinforcement), so frequently-recalled memories decay slowly and neglected ones fall below a retention floor and are dropped/down-weighted. The decision to forget vs reinforce is thus a pure function of access pattern + time, requiring no human review action. Demonstrated in the SiliconFriend companion chatbot.
- **Relevance to qmemd:** The cleanest attack on W4's manual-tax complaint: replace 'agent/human must run reviewed to reset the clock' with 'recall IS the reset' — the maintenance action and the use action become the same event. qmemd would add a last-recalled timestamp + recall-count to frontmatter and compute a decay-adjusted score; low-strength facts auto-populate the stale list with no review_by bookkeeping. This fits qmemd's existing 'staleness is surface-only' philosophy: down-rank/flag, don't delete. CATCH: time-decay is dangerous for a deliberately durable knowledge base — a credential, a hard project invariant, or a stated user preference must NOT fade just because nobody queried it for 180 days. qmemd's type system already encodes durability (user/feedback durable; project 90d, reference 180d), so decay must be type-scoped to project/reference and capped at down-ranking (never auto-expire), preserving the 'nothing auto-expires' contract. Also requires recording access events, which the current recall path doesn't persist.
- **Sources:** https://arxiv.org/abs/2305.10250; https://ar5iv.labs.arxiv.org/html/2305.10250; https://github.com/zhongwanjun/MemoryBank-SiliconFriend

#### Adaptive retrieval triggers — FLARE / Self-RAG / Adaptive-RAG (machine decides 'retrieve now')  ·  [W1, W2]  ·  🔬 research

- **What:** Three complementary mechanisms that move the 'should I retrieve?' decision off agent discipline and onto a machine signal: generation confidence, a learned reflection token, or a lightweight query-complexity classifier.
- **Mechanism:** FLARE (active RAG): while generating, if ANY token of the next candidate sentence has probability < threshold θ, treat the low-confidence span as a knowledge gap, form a query from it, retrieve, and regenerate; tuning fires retrieval on 40–80% of sentences. Self-RAG: the LM is trained to emit a 'Retrieve' REFLECTION TOKEN on demand (and critique tokens to grade the result) — retrieval becomes a learned, in-vocabulary action, not a prompted habit. Adaptive-RAG: a small T5-Large classifier predicts query complexity and ROUTES each query to one of no-retrieval / single-step / multi-step, matching always-retrieve accuracy at much lower cost.
- **Relevance to qmemd:** These are the literature's direct answers to W1/W2 — replacing 'the agent must remember to call recall' with a mechanism. For an MCP server that can't retrain the host model, the ADAPTIVE-RAG pattern is the feasible one: a cheap EXTERNAL classifier/heuristic over the user turn (or over the agent's first sentence) that decides whether to fire recall and auto-injects results — the trigger lives in a hook, not in prose. FLARE contributes a usable heuristic even without logprobs: fire recall when the agent EXPRESSES uncertainty / asks a question / hits an error (qmemd's own rules already say 'recall before diagnosing a build error' — that is a hand-written FLARE trigger; it could be detected mechanically instead of asked-for). CATCH: FLARE needs token logprobs and Self-RAG needs fine-tuning — both unavailable to a sidecar MCP server. Only the external-classifier route is viable, and it exposes qmemd's real architectural gap: it has a trigger NUDGE (the throttled PreToolUse beacon) but no on-turn trigger surface and no payload-injection channel. The decision logic is solved in the literature; qmemd needs the harness plumbing (a UserPromptSubmit-style hook that runs a classifier and prepends recall results) to use it.
- **Sources:** https://arxiv.org/abs/2305.06983; https://arxiv.org/abs/2310.11511; https://arxiv.org/abs/2403.14403; https://github.com/jzbjyb/FLARE; https://selfrag.github.io/

#### 2026 survey framework — write–manage–read lifecycle + six atomic ops + 'offline consolidation' frontier  ·  [W4, W5, W1]  ·  📄 paper

- **What:** Two current surveys give a vocabulary that locates qmemd's gaps precisely and name the emerging patterns for forgetting/conflict, including OFFLINE consolidation during idle periods (decouples maintenance from the write path and from agent discipline).
- **Mechanism:** 'Memory for Autonomous LLM Agents' (June 2026) formalizes a write(𝒰)–manage–read(ℛ) loop where write = summarize + dedup + priority-score + RESOLVE-CONTRADICTIONS + delete; read = retrieve + inject. 'Rethinking Memory in LLM-based Agents' (2505.00675) defines six atomic operations: Consolidation, Updating, Indexing, Forgetting, Retrieval, Condensation. Named conflict mechanisms: temporal versioning (prefer newest), source attribution (user statement > agent inference), contradiction detection (flagged, 'under-implemented'). Named decay: Ebbinghaus (MemoryBank), time-expiration, FIFO eviction, learned-forgetting (open). Emerging frontiers: principled OFFLINE consolidation ('hippocampal replay' during idle periods, dual-buffer probation), learning-to-forget under safety constraints, spaced repetition, and confidence-decay on reflections.
- **Relevance to qmemd:** Gives qmemd a map: it is strong on Indexing/Retrieval, partial on Updating (supersedes) and Forgetting (manual review_by), and ABSENT on Consolidation/Condensation. The single most aligned frontier is OFFLINE/IDLE consolidation: a background `qmemd consolidate` pass (cron / systemd-timer — service.ts already generates unit files) that dedups, merges near-duplicates, resolves the clear-cut conflicts, and synthesizes summaries OFF the critical path. This sidesteps the W5/W4 catch shared by Mem0/A-MEM (no model call on every write, no on-write silent mutation) and sidesteps W1/W2 (maintenance is scheduled, not agent-triggered) — it fits qmemd's git-backed batch model exactly. The survey's 'source attribution (user > inference)' and 'temporal versioning' are cheap deterministic conflict rules qmemd could apply before ever invoking an LLM. CATCH: it is a survey — patterns, not code; causal retrieval and learned forgetting are flagged open/under-implemented, and it explicitly warns that aggressive consolidation produces 'sanitized, generic' drift. Offline consolidation also needs a scheduler qmemd doesn't own end-to-end (leans on the harness/systemd it already touches via install-service).
- **Sources:** https://arxiv.org/html/2603.07670v1; https://arxiv.org/abs/2505.00675; https://arxiv.org/html/2505.00675v1

**Key takeaways:**

- The literature splits qmemd's W1/W2 (auto-trigger / discipline-dependence) into two solvable layers — a DECISION layer that is well-solved (FLARE token-confidence, Self-RAG learned tokens, Adaptive-RAG external classifier, MemGPT memory-pressure interrupts) and an INJECTION layer that is qmemd's real missing infrastructure. qmemd's beacon hook is a trigger nudge with no payload; the gap is an on-turn harness hook (UserPromptSubmit-style) that runs a cheap classifier and PREPENDS recall results into context. The decision logic is borrowable; the context-injection channel must be built.
- Adopt the Generative Agents 3-factor score (recency·importance·relevance, all weights 1, min-max normalized; recency = 0.995^hours, importance = 1–10 at write) to upgrade qmemd's single search score into a principled rank with self-prioritizing defaults — a soft mitigation of W1 that needs only an `importance` frontmatter field and last-access tracking.
- For W5, Mem0's ADD/UPDATE/DELETE/NOOP LLM memory-manager is the canonical auto-resolution design, and qmemd already owns the safe primitives to host it: supersedes/--replace (=UPDATE), soft-supersession (=DELETE without data loss), an authority score (=the gate), and git history (=the audit trail). The catch is philosophical not technical — qmemd deliberately 'never auto-resolves'; the surveys document the 'sanitized generic drift' failure mode, so any auto-merge must be reversible (soft-delete + audit commit) and gated by authority gap.
- For W4, the choice is decay-by-access (MemoryBank Ebbinghaus: recall reinforces, neglect fades — makes recall ITSELF the maintenance action, killing the review_by tax) vs synthesis (Generative Agents importance-sum reflection; survey 'offline consolidation'). Both must be TYPE-SCOPED: qmemd's durable user/feedback facts must never decay; only project(90d)/reference(180d) should, and only as down-ranking/flagging, preserving the 'nothing auto-expires' contract.
- OFFLINE / idle-time consolidation (a background `qmemd consolidate` via the systemd timer service.ts already emits) is the single best fit for qmemd's git-backed batch architecture: it gets the Mem0/A-MEM dedup+conflict+refresh benefits WITHOUT putting a model call on every write or risking on-write silent mutation, and it moves maintenance off agent discipline (W2/W4/W5) onto a schedule.
- A-MEM shows new facts can AUTO-REFRESH old ones (link generation via top-k cosine neighbors + LLM-decided bidirectional links + 'memory evolution' rewriting neighbor metadata); its note schema {keywords, tags, contextual-description, embedding, links} maps almost 1:1 onto qmemd frontmatter, making inter-fact links an incremental add that also improves retrieval defaults — but evolution-on-write is the costly/risky part and should be deferred to the offline pass.
- Cross-cutting catch: every auto-mechanism that helps W1/W4/W5 (importance rating, ADD/UPDATE/DELETE picking, A-MEM evolution) introduces a model call where qmemd writes are currently lex-only/no-model-load, and introduces silent-mutation risk into a store whose whole value is being a trustworthy durable record. The defensible path is: cheap deterministic rules first (temporal versioning, source attribution user>inference, type-scoped decay), model-based resolution only offline and only reversibly.

## Part 3 — synthesis: solutions mapped to weak spots

### Executive summary

The prior art is near-unanimous on qmemd's core gap (W1/W2): stop relying on the agent to call `recall` and move retrieval into a deterministic harness step. The proven, near-drop-in mechanism is a Claude Code UserPromptSubmit hook that runs a search on the submitted prompt and returns `hookSpecificOutput.additionalContext`, which the harness splices in before the model sees the turn (claude-mem, mann1x/claude-hooks, SuperBrain; the same idea as NeMo `auto_memory_agent` and MS Agent Framework Context Providers). qmemd already owns this surface — its `qmemd hook beacon` is a PreToolUse hook — but it only NUDGES; the upgrade is to INJECT the top-k facts, gated by the existing relevance floor and project scope. That single change also collapses W3 (rule bloat): claude-mem/mem0 ship almost no "remember to recall" prose because the hook enforces it, so qmemd.md can shrink to a trigger list. Heed claude-mem's v3→v4 scar: unconditional session-start dumps caused context pollution, so injection must be relevance-GATED, not always-on, and progressive-disclosure (inject an index of names/tags/dates, fetch bodies on demand) keeps the payload small. On W4, the cheap model-free borrows are usage-as-maintenance (MemoryBank: recall itself resets the review clock) and a Zep-style "current slice" filter that auto-excludes already-superseded facts from recall. On W5, the field actually VALIDATES qmemd's "surface, never auto-resolve" stance — mem0 v3 rolled back its LLM ADD/UPDATE/DELETE resolver to ADD-only because auto-delete measurably destroyed temporal context — so the defensible upgrades are non-lossy: filter superseded facts and offer opt-in, offline, git-reversible supersession, never a synchronous LLM merge on the write path.

### W1 — No component decides 'retrieve now.' recall is pull-only and agent-initiated; the snapshot is a static time-zero partial push and the beacon only emits a count+tag nudge. Every content-bearing retrieval depends on the LLM choosing to call recall.

**External solutions:**

- **claude-mem UserPromptSubmit hook (https://docs.claude-mem.ai/hooks-architecture, https://github.com/thedotmack/claude-mem)** — On every prompt a hook runs semantic search on the just-submitted prompt and returns the hits via hookSpecificOutput.additionalContext BEFORE the model processes the turn — retrieval becomes deterministic middleware, fail-open (exit 0, short timeout), not an LLM decision.
- **mann1x/claude-hooks (https://github.com/mann1x/claude-hooks)** — Reference impl of 'qmemd with the gaps fixed': per-prompt deterministic recall pipeline — HyDE query expansion, fan-out vector recall, attention/temporal-decay re-rank, cosine dedup, progressive-disclosure injection, all fail-open.
- **NeMo auto_memory_agent (https://docs.nvidia.com/nemo/agent-toolkit/latest/build-workflows/memory.html) + MS Agent Framework Context Providers (https://learn.microsoft.com/en-us/agent-framework/journey/adding-context-providers)** — A wrapper injects relevant memory before EVERY agent call (motivated literally by 'LLMs may forget to call memory tools'); rule: always-needed info = inject via provider, sometimes-needed = tool.
- **Adaptive-RAG / TARG (https://arxiv.org/abs/2403.14403, https://arxiv.org/pdf/2511.09803)** — A cheap EXTERNAL gate (classifier or training-free uncertainty threshold) decides whether to retrieve, outside the generator — the only adaptive-trigger family architecturally reachable by an MCP sidecar; cuts retrieval 70-90% vs always-RAG at equal accuracy.
- **Self-RAG / FLARE (https://arxiv.org/abs/2310.11511, https://arxiv.org/abs/2305.06983)** — Machine decides via reflection tokens / token-logprob confidence — the most elegant 'machine decides' answers, but OUT OF REACH: both need the generator's own tokens/logprobs, which qmemd never sees behind a closed Claude model.

**Recommendation:** Promote `qmemd hook beacon` (cli/qmemd.ts:486-520) into a UserPromptSubmit hook variant that runs a lex (CPU-cheap) recallQuery keyed on basename(cwd)+the user prompt and emits the top-k as hookSpecificOutput.additionalContext. Reuse the EXISTING relevance floor (belowFloor) + project scope as the injection gate so a weak query injects nothing (avoids claude-mem v3 context pollution). Keep it fail-open (exit 0, sub-second budget, reuse the 5s git timeout discipline). Use UserPromptSubmit specifically, not PreToolUse, because additionalContext was silently dropped on PreToolUse for a stretch (issues #19909, #19432).

- **Effort:** medium  ·  **Risk:** medium
- **Fits qmemd grain:** Strong fit: lex recall is model-free, the hook is best-effort/fail-open by construction, and markdown-file truth is untouched. Genuine limits: (1) it only covers the Claude Code surface — the CLI and other MCP clients still depend on manual recall, so this does not fully erase W1 system-wide; (2) per-turn lex recall adds latency and token cost and risks injecting noise, which is exactly why the belowFloor gate (not always-on push) is load-bearing.

### W2 — Correctness depends on the LLM remembering to call recall at the right instant. The machine cannot force it; the only levers today are prose rules in qmemd.md and a content-free nudge — pure discipline-dependence.

**External solutions:**

- **Cline/Roo-Code Memory Bank (https://docs.cline.bot/prompting/cline-memory-bank, https://github.com/GreatScottyMac/roo-code-memory-bank)** — NEGATIVE evidence: the most-popular markdown-memory pattern enforces loading purely by an emphatic prose instruction ('Cline MUST read ALL memory bank files at the start of EVERY task'). Its popularity does NOT validate prose-enforced loading — it has qmemd's exact W1/W2 flaw. Do not fix misses by lengthening rules.
- **Harness hooks vs CLAUDE.md (https://code.claude.com/docs/en/hooks, https://dev.to/sasha_podles/claude-code-using-hooks-for-guaranteed-context-injection-2jg)** — Hook-injected additionalContext is 'deterministic rather than advisory' — guaranteed on every matching event, unlike CLAUDE.md/Skills which the model may ignore. Author injected text as FACTUAL statements, not 'you MUST recall', or you trip prompt-injection defenses and the model surfaces it instead of acting.
- **Tool-description-as-prompt (https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools, https://github.com/boettiger-lab/mcp-data-server/issues/14)** — Put the WHEN-to-call triggers (positive AND negative) in the tool's own description. CAVEAT: Claude Code DEFERS MCP schemas — only tool names are visible until ToolSearch fetches them, so an agent that plans without first opening recall never sees the guidance; mitigate with a one-line 'fetch the recall schema before X' pointer in a session-start output.
- **MemGPT memory-pressure interrupt (https://arxiv.org/abs/2310.08560)** — The system injects a deterministic 'memory pressure' system message at a token threshold instead of trusting the agent — the canonical machine-side trigger. Not directly reachable (qmemd doesn't own Claude's context/control loop), but the principle is: the trigger lives in the harness, not the prompt.

**Recommendation:** Treat W2 as 'move the trigger out of prose into a deterministic component.' Ship the W1 hook-injection AND duplicate qmemd's existing trigger list ('before diagnosing a build/env error; on first-touch of a repo/tool; when the user references past context; when an instruction names a mechanism you're about to implement') verbatim into the recall MCP tool description as factual when/when-not statements, then delete those paragraphs from qmemd.md. Keep one irreducible trigger sentence in the always-loaded surface to defeat the deferred-schema blind spot.

- **Effort:** low  ·  **Risk:** low
- **Fits qmemd grain:** Fits: no model, no auto-mutation, MCP+CLI native. Tension: tool-description triggers are reliable only in clients that eagerly load schemas; in Claude Code's deferred-schema mode they have the same blind spot as the system prompt, so the description alone is not sufficient — the hook injection (W1) is what actually removes the discipline dependence; the description is the fallback for non-hook surfaces.

### W3 — The recall instructions are several long paragraphs. Long rules are brittle (rule fatigue): every word added to fix a miss makes the next miss likelier, and the footers/snapshot mitigations are themselves more discipline-dependent prose.

**External solutions:**

- **Anthropic Agent Skills three-level progressive disclosure (https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)** — Only name+description (~tens of tokens) stays in the system prompt; the body and references load on demand. Keep SKILL.md under ~5k words; for deterministic checks bundle a SCRIPT not prose ('code is deterministic; language interpretation isn't').
- **claude-mem / mann1x progressive-disclosure index (https://github.com/thedotmack/claude-mem)** — Inject an INDEX (titles, types, dates, token-counts, first line + char count) ~3-5x smaller than bodies; the model pulls full content on demand via MCP search/get only when needed.
- **GitHub Copilot copilot-instructions.md + applyTo glob (https://docs.github.com/en/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot) and Cursor rule tiers (https://cursor.com/docs/context/rules)** — One tiny always-injected core + glob-scoped extras that the SYSTEM (not the model) activates by file path. The glob is literally the on/off switch — scope injection to files in play instead of one always-on wall of rules.
- **Manus recitation / Anthropic harness feature-list (https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus, https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)** — Encode behavior as a structure the harness maintains; a 6-word recited checklist item in the recency zone beats three static rule paragraphs (but recitation keeps the GOAL salient, it doesn't fetch memory, and it costs actions).

**Recommendation:** Collapse qmemd.md to a one-line pointer + the bare WHEN-to-recall trigger list, and move all the HOW (scoping, snapshot, footer semantics, cross-project flags) into the existing qmemd-memory skill body / a references/recall.md that loads only during memory work. Switch the session snapshot's injected payload from fact bodies toward a progressive-disclosure INDEX (name/type/tags/date + 'recall <tag> for full text'), since qmemd already exposes MCP recall/get to fetch the body.

- **Effort:** low  ·  **Risk:** low
- **Fits qmemd grain:** Excellent fit — pure prose/format change, no engine change, leverages the skill + MCP get qmemd already ships. Tension: progressive disclosure is model-INVOKED (the skill body loads only if the model decides the description matches), which re-introduces W2 for the rule text itself — so the few always-on tokens must be spent on the WHEN (triggers), not the HOW. File-glob scoping (Copilot/Cursor) is attractive but qmemd facts are topical, not file-scoped, so it needs new metadata + a notion of 'files in context' the CLI lacks; defer it.

### W4 — Facts rot; review_by + the 'reviewed' command exist but impose a manual maintenance tax. recall ignores review_by, nothing auto-expires, and a stale fact silently outranks nothing — curation is a per-fact human chore.

**External solutions:**

- **MemoryBank Ebbinghaus decay (https://arxiv.org/abs/2305.10250, https://github.com/zhongwanjun/MemoryBank-SiliconFriend)** — Retention R = e^(-Δt/S), S = recall count; each recall increments S and resets Δt=0 — usage IS the review signal, so the manual 'reviewed' command becomes automatic for any fact actually in use. Down-rank, never auto-delete.
- **MemoryOS heat / Memary recency+frequency (https://arxiv.org/abs/2506.06326, https://github.com/kingjulio8238/memary)** — Heat = a·visits + b·engagement + c·recency drives demotion/eviction and auto-injects top-N — let access patterns, not calendar dates, set salience.
- **Zep/Graphiti bi-temporal (https://arxiv.org/abs/2501.13956, https://github.com/getzep/graphiti)** — Four timestamps (valid_at/invalid_at/created_at/expired_at); recall returns only the currently-valid slice so stale facts auto-drop from results without deletion and without a manual review pass — staleness is a property of the data model, not a chore.
- **Letta sleep-time compute / offline consolidation surveys (https://www.letta.com/blog/sleep-time-compute/, https://arxiv.org/html/2603.07670v1)** — A background agent runs during idle/SessionEnd to re-embed, dedup, refresh, and consolidate off the hot path — moves the maintenance tax to scheduled idle time; 'hippocampal replay' is the named frontier.

**Recommendation:** Two cheap, model-free borrows that preserve 'surface-only, nothing auto-expires': (a) make recallQuery forward-reset review_by (or increment a per-fact access-count/last-recalled field) on a hit — usage becomes the maintenance action, erasing the manual 'reviewed' tax for facts actually in use (markReviewed already does the forward-set, just wire it into the read path); (b) add an OFFLINE `qmemd consolidate` systemd-timer/SessionEnd pass (service.ts already emits unit files) that runs `doctor --fix` + the stale backlog + dedup PROPOSALS. Type-scope any decay: user/feedback stay durable, project(90d)/reference(180d) decay only as down-ranking/flagging, never eviction, and pinned/review_by:never are exempt.

- **Effort:** medium  ·  **Risk:** low
- **Fits qmemd grain:** Strong fit — both are time/metadata operations, no LLM in the write path, git-reversible, and they preserve the explicit 'nothing auto-expires' contract by capping at down-rank/flag. Tension: recording access events adds a WRITE on every recall (write-amplification + concurrency on a git-backed store) — keep it a lightweight sidecar counter, not a per-read git commit. Pure time-decay is dangerous for a durable KB (a once-a-year build fix would fade exactly when needed), so the type-scoping and pinned-exemption are non-negotiable.

### W5 — Near-duplicate and contradicting facts are detected (three-tier, Tier-2.5 contradiction classifier) and surfaced as disposition:'conflict', but never resolved. An undisciplined agent can --force past the signal or never re-run, so the corpus can still accrete contradictions and superseded facts still surface in recall.

**External solutions:**

- **mem0 ADD/UPDATE/DELETE/NOOP — AND its v3 rollback (https://arxiv.org/abs/2504.19413, https://docs.mem0.ai/migration/oss-v2-to-v3, https://github.com/mem0ai/mem0/pull/4903)** — An LLM picks one of four ops per write to auto-resolve. CRITICAL: mem0 v3 REVERTED to ADD-only because auto-DELETE/UPDATE measurably destroyed temporal context (+42.1 temporal, +23.1 multi-hop from the rollback) and the UPDATE prompt hallucinated old strings back in — the field's most-cited auto-resolver was abandoned. This validates qmemd's 'never silently resolve' stance.
- **Zep/Graphiti invalidate-don't-delete (https://getzep-graphiti.mintlify.app/concepts/temporal-model, https://blog.getzep.com/beyond-static-knowledge-graphs/)** — Resolution is a non-lossy state transition: set the old edge's invalid_at instead of deleting; recall filters to the valid slice by default. Maps almost 1:1 onto qmemd's existing supersedes/superseded_by frontmatter.
- **A-MEM memory evolution (https://arxiv.org/abs/2502.12110)** — A new note auto-links to neighbors by cosine similarity and can refine their tags/description rather than spawn a near-dup — consolidate by enriching, not deleting. Costs an LLM call per write and risks silent drift; adopt the linking, not the unsupervised rewrite.

**Recommendation:** Keep surface-by-default (vindicated by mem0 v3). Ship the cheap non-lossy win first: auto-EXCLUDE facts carrying superseded_by from the default recall slice (Zep 'current slice'), with an --include-superseded escape hatch — qmemd already stores the field but only DISPLAYS it. Then, optionally, an OPT-IN, OFFLINE (consolidate-pass) supersede-on-high-confidence-contradiction that reuses the existing authority comparison: it SETS superseded_by (soft, reversible, git-committed, writes a .bak like doctor) and never hard-deletes. Map any 'DELETE' to soft-supersede; reserve genuine/low-authority-gap conflicts for human surfacing.

- **Effort:** low  ·  **Risk:** low
- **Fits qmemd grain:** The superseded-filter fits cleanly and is NOT a violation of 'never auto-resolve' — the agent already set superseded_by; recall is just respecting it. The auto-supersede step is the genuine TENSION: it crosses qmemd's explicit 'engine never auto-resolves' line, so it must stay opt-in, offline (not on the model-free write path), authority-gated, and git-reversible — i.e. never silent, never synchronous, never lossy. mem0's own retreat is the strongest argument NOT to put an LLM resolver on the write path; qmemd's git-commit-before-reindex + .bak give it the audit/undo safety net that makes the conservative offline version defensible.

### Top actions (ranked)

1. (W1/W2/W3, highest leverage) Convert `qmemd hook beacon` from nudge to INJECTION via a UserPromptSubmit hook: run a lex recallQuery on basename(cwd)+prompt, gate by the existing belowFloor relevance floor + project scope, emit top-k as hookSpecificOutput.additionalContext, fail-open (exit 0). 3. (W5, cheap & safe) Auto-exclude superseded_by facts from the default recall slice (Zep 'current slice'), with an --include-superseded flag. Pure metadata filter, no model, already-present frontmatter — turns an unused display field into real curation.

4. (W4, cheap & model-free) Wire recall-as-maintenance: a recall hit forward-resets review_by / increments an access-count (MemoryBank spacing effect), erasing the manual 'reviewed' tax for facts in active use. Type-scope and exempt pinned/durable.

6. (W3) Switch the session-snapshot injected payload toward a progressive-disclosure INDEX (name/type/tags/date + 'recall <tag>'), not bodies, fetching full text via the existing MCP get — shrinks per-turn tokens while keeping everything reachable.

### Patterns to steal

- Hook injection contract: UserPromptSubmit/SessionStart hook returns hookSpecificOutput.additionalContext, fail-open (exit 0, short timeout) so a memory outage never blocks the turn; phrase injected facts as FACTUAL statements ('This repo uses X'), not imperative 'you MUST recall', to avoid tripping prompt-injection defenses (claude-mem; Claude Code hooks docs).
- Relevance-gated push, not always-push: use qmemd's existing belowFloor + project scope as the injection gate (claude-mem v3 context-pollution scar) — the lesson is gate, don't dump.
- Progressive-disclosure index: inject name/type/tags/date/first-line + char-count, fetch bodies on demand via MCP get (~3-5x smaller payload) (claude-mem, mann1x).
- Zep 'current slice' filter: recall returns only facts whose invalid_at/superseded_by is null/unset by default; map a contradiction to set the old fact's superseded_by (invalidate, don't delete), keeping history (Graphiti four-timestamp model: valid_at/invalid_at/created_at/expired_at).
- MemoryBank spacing effect: on a recall hit, increment strength S and reset Δt; retention R = e^(-Δt/S); never-recalled + old ⇒ flag as decay candidate (down-rank, never auto-evict; type-scope to project/reference, exempt pinned).
- Generative Agents 3-factor rank: score = a·recency + b·importance + c·relevance, all min-max normalized to [0,1], reference weights relevance≈3/importance≈2/recency≈0.5, recency = 0.995^hours-since-last-access — upgrades qmemd's single search score into a self-prioritizing rank (needs only an importance frontmatter field + last-access tracking).
- mem0 4-op decision TABLE (ADD/UPDATE/DELETE/NOOP) as a SUGGESTED, human/offline-confirmed disposition layered on qmemd's existing authority comparison — never silent application; and heed mem0 v3's 'accumulate + rank-current-first' as the safer alternative to merge/overwrite.
- Tool-description-as-prompt structure: sentence1 what / sentence2 when-to-use triggers / sentence3 when-NOT / sentence4 returns — put recall's trigger list here; but note Claude Code defers MCP schemas (mcp-data-server #14), so keep one irreducible trigger sentence in an always-loaded surface.
- Adaptive-RAG/TARG external gate: a cheap model-free heuristic over the turn (qmemd's beacon already computes first-touch + repo-pivot + build-error signals) decides whether to fire injection — deterministic trigger > prose rule, and a 'no new facts ⇒ stay silent' guard prevents pollution.
- Offline/idle consolidation ('hippocampal replay'): a scheduled pass (systemd timer via service.ts) that runs doctor --fix + dedup + supersession proposals where being wrong is cheap and git-reversible — keeps the model out of the write path (W4/W5 surveys; Letta sleep-time).
